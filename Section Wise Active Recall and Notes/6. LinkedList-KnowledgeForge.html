Linked List

Dynamic node-chain · Non-contiguous memory · O(1) insert/delete at ends

💡 A linked list is a chain of nodes. Each node holds a data value and a pointer to the next node. Nodes live at random memory addresses — connected by references, not physical proximity.

📐 Why Linked List?

Problem with ArraysHow Linked List Solves It
Fixed size — must know size at creationDynamic — nodes created on demand
Needs contiguous memory blockNodes scattered anywhere in RAM
Inserting/deleting in middle requires shifting all elements → O(n)Change two pointers → O(1) once position is found
Resizing creates an entirely new arrayJust add or remove a node

🗺️ Memory Reality — Nodes Are Scattered

📌 Unlike arrays, linked list nodes do NOT need to be adjacent in memory. Each node can live at a completely random address. The pointer field stores the address of the next node, creating a logical chain across scattered locations.

🔑 Core Rules

  • Never lose the HEAD reference — losing head means losing access to the entire list. Always use a temporary pointer for traversal.
  • TAIL reference is optional — having it makes addLast() O(1) instead of O(n). Without it, you must traverse to find the end.
  • Traversal only goes forward — in a singly linked list you cannot go backward. To delete the last node you must traverse to the second-to-last.
  • Last node always points to null — this is the termination condition for all traversal loops.

🚂 The Train Analogy — Think of a linked list as a train. Each carriage = node. Each carriage knows its own value AND which carriage comes next. HEAD = the engine (first car). TAIL = the last car. null = end of track — no more carriages.
M01 AddFirst(T value) O(1) Built-in
🧒 ELI5 — What does it do?
Adds a new value at the very beginning of the list. Whatever was first before is now second.
🚂 You attach a new engine car to the front of the train. The old front car becomes car #2. No other car moves — just the "who is first?" label changes to the new car.

How it works — pointer by pointer

1
Node newest = new Node(7, null);
Create a brand-new node. Data = 7. Next pointer = null (pointing nowhere yet).
2
if (list is empty) → First = Last = newest
Special case: the very first item ever added is both head AND tail simultaneously.
3
newest.Next = First; (new node → old first)
Point the new node's Next arrow at whatever used to be first. The chain is preserved.
4
First.Previous = newest; (doubly-linked backward link)
C#'s built-in is doubly-linked, so the old first node also gets a "look-back" arrow to the new node.
5
First = newest; Count++;
Move the HEAD label to the new node. Increment counter. Done.

M02 AddLast(T value) O(1) Built-in
🧒 ELI5 — What does it do?
Adds a new value at the very end of the list. Everything else stays in place.
🚂 You hook a new cargo car onto the back of the train. The old last car now links forward to the new one. The TAIL label moves to the new car.

How it works — pointer by pointer

1
Node newest = new Node(value, null);
New node, Next=null — it will be the last, so nothing comes after it yet.
2
if (empty) → First = Last = newest
First item — it is simultaneously head and tail.
3
Last.Next = newest; (old tail → new node)
The old last node's forward arrow points to the new node.
4
newest.Previous = Last; (doubly-linked backward link)
New node looks back at the old tail.
5
Last = newest; Count++;
Move the TAIL label to the new node.
📌 Why O(1) and not O(n)? Because we keep a direct Last pointer. Without it, we'd walk every single node to reach the end — O(n). The TAIL pointer is the secret that makes append fast.
csharp
var list = new LinkedList<int>();
list.AddLast(7);    // list: 7             First=Last=node(7)
list.AddLast(4);    // list: 7 → 4         Last moves to node(4)
list.AddLast(12);   // list: 7 → 4 → 12   Last moves to node(12)

Console.WriteLine(list.Last.Value);  // 12  ← tail
Console.WriteLine(list.Count);       // 3
TimeO(1) alwaysSpaceO(1)
M03 AddBefore(node, value) / AddAfter(node, value) O(1) Built-in
🧒 ELI5 — What does it do?
You already have your hand on a specific carriage. AddBefore squeezes a new car in front of it. AddAfter squeezes one in behind it. Only 4 pointer wires are changed — nothing else moves.
🔗 Like cutting a chain at one link and inserting a new link in the gap. You need to hold the exact spot (node reference) — if you only knew "it's the 3rd one," you'd need to walk there first (O(n)), but once you're holding it, the insert is instant (O(1)).

AddBefore — pointer steps

1
Create newest node
Fresh node, both pointers null for now.
2
newest.Next = node; newest.Previous = node.Previous;
New node's forward arrow → target. New node's backward arrow → predecessor.
3
node.Previous.Next = newest;
Predecessor's forward arrow → new node (was pointing at target before).
4
node.Previous = newest;
Target's backward arrow → new node.
5
if (node was First) → First = newest; Count++;
If we inserted before the head, update HEAD.

M04 RemoveFirst() / RemoveLast() O(1) built-in Built-in
🧒 ELI5 — What does it do?
RemoveFirst() — detach the front car. The 2nd car becomes the new front.
RemoveLast() — detach the last car. The 2nd-to-last becomes the new tail.
⚠️ Critical edge case: if the list has only ONE item and you remove it, both HEAD and TAIL must become null. Forgetting to null the tail when the list empties is a classic bug — the tail keeps pointing at a deleted node (dangling reference).

RemoveFirst — pointer steps

!
if (Count == 0) throw InvalidOperationException
Can't remove from nothing. The built-in throws — always check Count > 0 first!
1
First = First.Next;
HEAD label jumps forward to the 2nd node. Old first node is now unreachable → garbage collected.
2
if (First != null) First.Previous = null;
Clear the new head's backward arrow — nothing is before it now.
3
if (First == null) Last = null;
⚡ CRITICAL: list is now empty — TAIL must also be null. Both HEAD and TAIL must be null together.
4
Count--;
Decrement the counter.

Why RemoveLast is O(1) in built-in but O(n) in custom singly-linked

✅ C# Built-in (doubly-linked)
Last.Previous directly holds the 2nd-to-last node.
Just do: Last = Last.Previous; Last.Next = null;
→ O(1)
⚠️ Custom singly-linked
No backward pointer exists. Must walk from HEAD all the way to position (size-1) counting steps.
→ O(n)
csharp
var list = new LinkedList<int>(new[] { 7, 4, 12, 8, 3 });

list.RemoveFirst();  // removes 7  → list: 4 → 12 → 8 → 3
list.RemoveLast();   // removes 3  → list: 4 → 12 → 8

Console.WriteLine(list.First.Value); // 4
Console.WriteLine(list.Last.Value);  // 8
Console.WriteLine(list.Count);       // 3

// ⚠️ Always guard — built-in THROWS on empty list:
if (list.Count > 0) list.RemoveFirst();  // safe
// list.RemoveFirst() on empty → InvalidOperationException!
Built-in both opsO(1)Custom RemoveLastO(n)
M05 Remove(T value) / Remove(LinkedListNode<T> node) O(n) / O(1) Built-in
🧒 ELI5 — Two versions, very different speeds
Remove(value): "Find the car labelled '30' somewhere in this 1000-car train and remove it." → Must walk every car from the front → O(n).

Remove(node): "I'm already standing next to car 30, holding it. Remove it." → Just unhook 4 wires → O(1).
🔍 Difference: "find my friend in this crowd" (O(n)) vs "my friend is right here" (O(1)). The node reference IS the "my friend is right here" part.

Remove(node) — what changes (O(1) path)

1
node.Previous.Next = node.Next;
The car BEFORE our target now skips over it and links to the car AFTER it.
2
node.Next.Previous = node.Previous;
The car AFTER our target now looks back to the car BEFORE it (skipping target).
3
if (node == First) First = node.Next;
If we removed the head, move HEAD forward.
4
if (node == Last) Last = node.Previous;
If we removed the tail, move TAIL backward.
5
node.List = null; Count--;
Detach node from this list (marks it as "free"). Decrement counter.

M06 Find(T) / FindLast(T) / Contains(T) O(n) Built-in
🧒 ELI5 — What do they do?
Find(30): Walk through every car from the front. At each car, look at its label. If label = 30, hand me that car (the node object itself — not just the label).

FindLast(30): Same but starts from the BACK — useful when duplicates exist and you want the last occurrence.

Contains(30): Just yes/no — is there any car labelled 30? (Internally just calls Find and checks if null.)

How Find() walks the list

1
Node current = First;
Start at the front of the list.
2
while (current != null)
Keep going until we fall off the end (null = no more nodes).
3
if (current.Value equals target) → return current;
Match found! Return the WHOLE NODE — not just the value. This is key: the node is usable for further O(1) ops.
4
current = current.Next; (move forward)
Check next node.
5
return null; (fell off end — not found)
Value is not in the list. ALWAYS null-check the result!
💡 The superpower of Find(): It returns a LinkedListNode<T>. Use that node with AddBefore(), AddAfter(), or Remove(node) to do more work in O(1) — without searching again. Classic pattern: Find once, operate many times.
csharp
var list = new LinkedList<string>(new[] { "Alice", "Bob", "Bob", "Carol" });

// Find() — returns node object or null
LinkedListNode<string> first = list.Find("Bob");
Console.WriteLine(first?.Value);       // "Bob" (first occurrence)
Console.WriteLine(first?.Next?.Value); // "Bob" (can navigate from it!)

// FindLast() — starts from tail (doubly-linked advantage)
LinkedListNode<string> last = list.FindLast("Bob");
// last points to the SECOND "Bob"

// Contains() — just true/false
bool hasBob = list.Contains("Bob");   // true
bool hasZoe = list.Contains("Zoe");   // false

// ⚠️ Not found = null! Always check:
LinkedListNode<string> miss = list.Find("Zoe");
if (miss != null) list.AddAfter(miss, "After");  // safe pattern
All threeO(n) — full scan worst caseSpaceO(1)
M07 Clear() · CopyTo(array, index) · foreach / GetEnumerator() O(n) Built-in
🧒 ELI5 — What do they do?
Clear(): Wipe the entire train. Every car, gone. Count = 0.

CopyTo(array, i): Pour all the values from the linked list into a regular array starting at position i. Like unloading all cargo into a warehouse row.

foreach: Walk every car one by one from front to back — the normal way to visit all values.

Why is Clear() O(n) and not O(1)?

You might think "just set First=null and Last=null and it's done in O(1)." But C# must visit every node to detach each node's List reference — so the garbage collector knows each object is truly freed. Without this detachment, the GC can't collect them properly. That visit = O(n).

csharp
var list = new LinkedList<int>(new[] { 7, 4, 12, 8, 3 });

// ── foreach — easiest way to visit all values ─────────────
foreach (int val in list)
    Console.Write(val + " ");
// Output: 7 4 12 8 3

// ── Node traversal — gives you the node objects ────────────
// Use this when you need .Next / .Previous / the node itself
for (var n = list.First; n != null; n = n.Next)
    Console.Write(n.Value + " ");
// Output: 7 4 12 8 3

// ── CopyTo — dump into a regular array ────────────────────
int[] arr = new int[list.Count]; // pre-allocate exact size!
list.CopyTo(arr, 0);             // fills from index 0
// arr = [7, 4, 12, 8, 3] — now you can binary search etc.

// ── Clear ─────────────────────────────────────────────────
list.Clear();
Console.WriteLine(list.Count);  // 0
Console.WriteLine(list.First);  // null
ClearO(n)CopyToO(n)foreachO(n) per pass
C01 addAny(int element, int position) — Custom O(n) Course Custom
🧒 ELI5 — What does it do?
Insert a new value at position number N (1-based). The computer walks from the front counting steps, stops at position N-1, then carefully squeezes the new node in between N-1 and N.
🚂 "Insert a new car at position 3." Walk from the engine, count 2 cars, stop. Unhook car 2 from car 3. Hook car 2 → new car → car 3. The rest of the train is untouched.

Step by step — the two-pointer write order is CRITICAL

!
if (position ≤ 0 || position ≥ size) → print "Invalid", return
Position must be a valid middle slot. Ends are handled by addFirst/addLast.
1
Node newest = new Node(element, null);
Create the new node. Next = null for now.
2
Node p = head; int i = 1;
Start at head, counter = 1.
3
while (i < position - 1): p = p.next; i++;
Walk to the node BEFORE the insertion point. Stop one early — we need to rewire FROM that predecessor.
4
newest.next = p.next; ← DO THIS FIRST ✅
Point new node forward to whatever comes after p. This SAVES the rest of the chain before we overwrite anything.
5
p.next = newest; ← DO THIS SECOND ✅
Now p points to the new node. Chain is: p → newest → (rest of list).
6
size++;
One more item.
⚠️ If you swap steps 4 and 5 — catastrophe. Doing p.next = newest FIRST overwrites your only reference to the rest of the list. Then newest.next = p.next just points newest back to itself. The entire tail is gone forever — unreachable, leaked memory.
csharp
// List: 7 → 4 → 12 → 8 → 3  (size=5)
// addAny(99, 2) — insert 99 at position 2 (between 7 and 4)

// Walk trace:
//   p=head(7), i=1
//   Loop condition: i < position-1 = 1 → loop doesn't run
//   p is at node(7)
//   newest.next = p.next = node(4)   ← new → 4
//   p.next = newest                  ← 7 → 99
//   Result: 7 → 99 → 4 → 12 → 8 → 3

public void addAny(int e, int position)
{
    if (position <= 0 || position >= size)
    { Console.WriteLine("Invalid Position"); return; }

    Node newest = new Node(e, null);
    Node p = head;
    int i = 1;
    while (i < position - 1)  // walk to node BEFORE target
    { p = p.next; i++; }

    newest.next = p.next;  // ✅ FIRST: save rest of chain
    p.next = newest;        // ✅ SECOND: link predecessor to new node
    size++;
}
TimeO(n) worst caseSpaceO(1)
C02 removeAny(int position) — Custom O(n) Course Custom
🧒 ELI5 — What does it do?
Remove the car at position N from the train. Walk to car N-1 (one before it), save car N's value, then make car N-1 skip directly to car N+1. Car N is disconnected and disappears.

Step by step

!
Validate position
Must be a valid middle position.
1
Walk to node at (position - 1)
Stop at the predecessor — we need to rewire FROM it, not from the target itself.
2
int e = p.next.element;
Save the target's value BEFORE we lose the reference.
3
p.next = p.next.next;
Predecessor now skips over the target → points directly to what was after target. Target is now unreachable → GC collects it.
4
size--; return e;
Counter down. Return the removed value.
csharp
// List: 7 → 4 → 12 → 8 → 3  (size=5)
// removeAny(3) — remove node at position 3 (value=12)

// Walk trace:
//   p walks to position 2 (node with value=4)
//   e = p.next.element = 12        ← save it
//   p.next = p.next.next = node(8) ← 4 now points to 8
//   Result: 7 → 4 → 8 → 3

public int removeAny(int position)
{
    if (position <= 0 || position >= size - 1)
    { Console.WriteLine("Invalid Position"); return -1; }

    Node p = head;
    int i = 1;
    while (i < position - 1)  // walk to node BEFORE target
    { p = p.next; i++; }

    int e = p.next.element;    // save target's value
    p.next = p.next.next;      // bypass the target node
    size--;
    return e;
}
TimeO(n)SpaceO(1)
C03 insertSorted(int element) — Custom O(n) Course Custom
🧒 ELI5 — What does it do?
The list is already in sorted order (e.g. 3 → 7 → 12). You want to add 8 and keep it sorted. Walk through until you find the right gap (after 7, before 12), and insert there.
🃏 Like holding a sorted hand of cards and sliding a new card into its correct position — you scan left to right until you find where it fits, then squeeze it in.

Why two pointers (p and q)?

In a singly-linked list you can only move forward. Once p overshoots the target position (lands on 12 when inserting 8), you can't go back. So you keep q trailing one step behind p — when p overshoots, q is at the correct predecessor.

1
p = head; q = head;
Both pointers start at the front.
2
while (p != null AND p.element < e): q = p; p = p.next;
Advance while p's value is smaller than e. q trails one behind. Stop when p lands on a value ≥ e — that's where e goes BEFORE p.
3
if (p == head) → insert before head, update HEAD
Special case: new value is smaller than everything — it goes at the very front.
4
else: newest.next = q.next; q.next = newest;
Insert between q and p. Same two-step rule: forward link first, then backward link.

Trace: insertSorted(8) into [3 → 7 → 12]

Init: p=3, q=3
p.elem(3) < 8 → q=3, p=7
p.elem(7) < 8 → q=7, p=12
p.elem(12) ≥ 8 → STOP. q=7, p=12
p ≠ head → insert between q(7) and p(12)
newest.next = q.next(=12) → newest.next = 12
q.next = newest → 7 → newest(8)
Result: 3 → 7 → 8 → 12 ✓
csharp
LinkedList l = new LinkedList();
l.insertsorted(7);   // list: 7
l.insertsorted(4);   // list: 4 → 7       (4 < 7, goes front)
l.insertsorted(12);  // list: 4 → 7 → 12
l.insertsorted(8);   // list: 4 → 7 → 8 → 12
l.insertsorted(3);   // list: 3 → 4 → 7 → 8 → 12  (smallest!)
l.display();
// Output: 3 --> 4 --> 7 --> 8 --> 12 -->

🔷 LinkedList Class — Full Implementation

csharp
class LinkedList
{
    private Node head;   // first node
    private Node tail;   // last node (for O(1) addLast)
    private int size;    // number of elements

    public LinkedList() { head = null; tail = null; size = 0; }

    public int  length()  => size;
    public bool isEmpty() => size == 0;

    // ── Add ──────────────────────────────────────────────────
    public void addLast(int e)
    {
        Node newest = new Node(e, null);
        if (isEmpty()) head = newest;
        else           tail.next = newest;
        tail = newest;
        size++;
    }

    public void addFirst(int e)
    {
        Node newest = new Node(e, null);
        if (isEmpty()) { head = newest; tail = newest; }
        else           { newest.next = head; head = newest; }
        size++;
    }

    public void addAny(int e, int position)
    {
        if (position <= 0 || position >= size) { Console.WriteLine("Invalid Position"); return; }
        Node newest = new Node(e, null);
        Node p = head; int i = 1;
        while (i < position - 1) { p = p.next; i++; }
        newest.next = p.next;
        p.next = newest;
        size++;
    }

    public void insertsorted(int e)
    {
        Node newest = new Node(e, null);
        if (isEmpty()) { head = newest; size++; return; }
        Node p = head, q = head;
        while (p != null && p.element < e) { q = p; p = p.next; }
        if (p == head) { newest.next = head; head = newest; }
        else           { newest.next = q.next; q.next = newest; }
        size++;
    }

    // ── Remove ───────────────────────────────────────────────
    public int removeFirst()
    {
        if (isEmpty()) { Console.WriteLine("List is Empty"); return -1; }
        int e = head.element;
        head = head.next;
        size--;
        if (isEmpty()) tail = null;
        return e;
    }

    public int removeLast()
    {
        if (isEmpty()) { Console.WriteLine("List is Empty"); return -1; }
        Node p = head; int i = 1;
        while (i < size - 1) { p = p.next; i++; }
        tail = p;
        p = p.next;
        int e = p.element;
        tail.next = null;
        size--;
        return e;
    }

    public int removeAny(int position)
    {
        if (position <= 0 || position >= size - 1) { Console.WriteLine("Invalid"); return -1; }
        Node p = head; int i = 1;
        while (i < position - 1) { p = p.next; i++; }
        int e = p.next.element;
        p.next = p.next.next;
        size--;
        return e;
    }

    // ── Search / Display ─────────────────────────────────────
    public int search(int key)
    {
        Node p = head; int index = 0;
        while (p != null)
        {
            if (p.element == key) return index;
            p = p.next; index++;
        }
        return -1;
    }

    public void display()
    {
        Node p = head;
        while (p != null) { Console.Write(p.element + " --> "); p = p.next; }
        Console.WriteLine();
    }
}

🔷 Key Design Decisions Explained

DecisionWhy
Maintain a tail pointerMakes addLast() O(1) instead of O(n). Without it you'd traverse every time.
Maintain a size counterMakes length() O(1). Without it you'd need to traverse to count nodes.
Check isEmpty() before removePrevents null pointer exceptions on an empty list.
Two-step pointer update in addAnynewest.next = p.next BEFORE p.next = newest — reversal loses the chain.
Two pointers (p, q) in insertSortedq trails one behind p so we know where to insert when p overshoots.

🔑 The Big Insight

📌 The critical difference between singly and doubly linked lists is RemoveLast: O(n) in singly (must find second-to-last), O(1) in doubly (use tail.Previous). This single operation drives the choice between them in production code. C#'s built-in uses doubly-linked for exactly this reason.

📚 Fundamentals

Q01DEFWhat are the two fields every linked list node contains? What does each store?
Answer
Every node contains: Data field — stores the actual value (int, string, object, etc.). Pointer field (Next) — stores the memory address (reference) of the next node in the chain. In a doubly-linked list, a third field Previous stores the address of the preceding node.
C#: public int element; public Node next;
Q02DEFWhat is the HEAD pointer and why must you never lose it?
Answer
HEAD is a reference to the first node of the linked list. It is the only entry point — there is no index-based access. If you overwrite or lose HEAD without saving it, you have no way to reach any node in the list. The entire list becomes unreachable and gets garbage-collected. Always use a temporary pointer for traversal: Node p = head; never head = head.next while iterating.
Q03WHYWhy do we maintain a TAIL pointer in the custom implementation? What is the cost of not having it?
Answer
Without a TAIL pointer, every addLast() would require traversing the entire list from HEAD to find the end → O(n) per insertion. Maintaining a direct TAIL pointer makes addLast() O(1): simply wire the current tail's next to the new node and update tail. This is a classic space-time tradeoff — one extra pointer saves O(n) time per append.
Q04WHYLinked lists do NOT need contiguous memory. Why does this matter?
Answer
Arrays require a single contiguous block of memory equal to the full array size. If your system has 1GB RAM but it's fragmented into 100MB chunks, you cannot create a 200MB array — even though there's enough total RAM. A linked list can use any available memory fragments because each node is independently allocated. This makes linked lists better suited for dynamic, unbounded data in memory-constrained environments.
Q05COMPAREWhat is the key structural difference between a singly-linked list (course custom) and C#'s built-in LinkedList<T>?
Answer
Custom (singly): Each node has only a next pointer. Can only traverse forward. RemoveLast() is O(n) because you must find the second-to-last node.

C# built-in (doubly): Each LinkedListNode<T> has Next AND Previous. Can traverse both directions. RemoveLast() is O(1) via Last.Previous. Every operation that only requires node rewiring becomes O(1) once you have the node reference.

Q07GOTCHAIn addAny(), what is the critical order for the two pointer assignment statements? What happens if you reverse them?
Answer
The correct order is: first newest.next = p.next, then p.next = newest.

If reversed: p.next = newest overwrites the only reference to the rest of the list. newest.next = p.next would then point newest back to itself (since p.next is now newest). The entire tail of the list is lost — unreachable, leaked memory.
Q08GOTCHARemoveLast() is O(1) in C#'s built-in but O(n) in the custom singly-linked implementation. Why?
Answer
In the custom singly-linked list, nodes only have a next pointer. To remove the last node, you need to set tail = second-to-last node — but there's no backward pointer to find it. You must traverse from HEAD counting n-1 steps → O(n).

In C#'s doubly-linked built-in: Last.Previous directly points to the second-to-last node. Removal is just: Last = Last.Previous; Last.Next = null → O(1).
Q09APPLYUsing the C# built-in, insert value 25 between nodes 20 and 30 without traversing the list again after finding 30.

// Result: 10 → 20 → 25 → 30 → 40
Pattern: Find once O(n), then all local operations (AddBefore, AddAfter, Remove(node)) are O(1).

List: 3 → 7 → 12, inserting 8.

  2. p.element(3) < 8 → q=3, p=7

Q12APPLYWhat does Find() return if the value is not in the list? How does this differ from Contains()?
Answer
Find() returns null when the value is not found. Contains() returns false. Internally, Contains(value) is equivalent to Find(value) != null. Use Find() when you need the node object for subsequent O(1) insertion/deletion. Use Contains() when you only need a boolean answer.

⏱️ Complexity Questions

Q13WHYWhy is random access (getting element at index i) O(n) in a linked list but O(1) in an array?
Answer
In an array, element at index i is at memory address base + i × elementSize — calculated in one step. In a linked list, there is no address formula. You must start at HEAD and follow i pointers: head.next.next...next (i times). The i-th node could be anywhere in memory. This is the fundamental tradeoff: linked lists sacrifice O(1) random access for O(1) insert/delete at known positions.
Q14COMPAREFill in: AddFirst is O(___), RemoveLast in singly-linked is O(___), Find is O(___), AddAfter(node) is O(___).
Answer
  • AddFirst: O(1) — only rewire head
  • RemoveLast (singly): O(n) — traverse to second-to-last
  • Find: O(n) — linear scan
  • AddAfter(node): O(1) — given the node, just rewire neighbors
Q15WHYSpace complexity of a linked list is O(n). But it uses MORE memory per element than an array. Explain this apparent contradiction.
Answer
Both are O(n) because both scale linearly with n. But the constant factor differs: an array stores just the values (4 bytes per int). A singly-linked list stores value + 1 pointer (~12 bytes per int node). A doubly-linked list stores value + 2 pointers + list-ref (~32 bytes per node). O(n) hides this constant — in practice a linked list uses 3–8× more memory per element than an array. The O(n) is the same class, but the real memory is significantly higher.

🟢 Interview Questions

Q16INTERVIEWHow would you detect if a linked list has a cycle?
Answer — Floyd's Cycle Detection
Use two pointers: slow (moves 1 step) and fast (moves 2 steps). If there's a cycle, fast will eventually lap slow and they'll meet at the same node. If fast reaches null, no cycle.
Code: var slow = head; var fast = head;
while (fast != null && fast.Next != null) {
  slow = slow.Next; fast = fast.Next.Next;
  if (slow == fast) return true;
}
return false;
Time: O(n) · Space: O(1)
Q17INTERVIEWHow would you find the middle of a linked list in a single pass?
Answer — Two Pointer Technique
Use slow (1 step) and fast (2 steps). When fast reaches the end, slow is at the middle.
Code: var slow = head; var fast = head;
while (fast != null && fast.Next != null) {
  slow = slow.Next; fast = fast.Next.Next;
}
// slow is now at the middle node
Time: O(n) · Space: O(1)
Q18INTERVIEWHow would you reverse a singly linked list in-place?
Answer — Three Pointer Reversal
Use three pointers: prev (null), current (head), next (null). At each step: save next, reverse current's pointer, advance both.
Code: Node prev = null, curr = head, next = null;
while (curr != null) {
  next = curr.next;
  curr.next = prev;
  prev = curr;
  curr = next;
}
head = prev;
Time: O(n) · Space: O(1)
Q19INTERVIEWHow do you find the k-th node from the end of a linked list in a single pass?
Answer — Two Pointer Gap Technique
Put two pointers k steps apart. Advance both together until the fast one hits null — slow is at k-th from end.
Code: var fast = head;
for (int i = 0; i < k; i++) fast = fast.Next;
var slow = head;
while (fast != null) { slow = slow.Next; fast = fast.Next; }
// slow.element is the k-th from end
Time: O(n) · Space: O(1)
Q20INTERVIEWWhen would you choose a linked list over an array in production code? When would you choose an array?
Answer
Choose LinkedList when:
  • Frequent insertions/deletions at the front or known positions (O(1) vs array's O(n) shift)
  • Unknown or highly variable size — no wasted pre-allocated space
  • Implementing a queue or deque where both ends are used
  • Memory is fragmented — can't allocate one large contiguous block
Choose Array when:
  • Need random access by index — O(1) vs LL's O(n)
  • Frequent iteration over all elements — cache locality is 5–10× faster
  • Size is known and stable — no allocation overhead
  • Binary search needed — requires random access
Q21INTERVIEWWhat is the difference between Remove(T value) and Remove(LinkedListNode<T> node)? Which is O(1) and why?
Answer
Remove(T value) is O(n) — it must first scan the list to find the node with that value, then remove it.

Remove(LinkedListNode<T> node) is O(1) — you already have the node reference. It directly rewires: node.Previous.Next = node.Next and node.Next.Previous = node.Previous. This is the key advantage of a doubly-linked list: once you have a node reference, removal is constant time.
Q22INTERVIEWExplain the two-pointer (p, q) technique used in insertSorted(). Why is one pointer not enough?
Answer
In a singly-linked list, once p moves past the insertion point, you can't go back (no previous pointer). You need q to trail one step behind p so that when p.element ≥ e (overshoot), q is the node before the insertion point. Insert between q and p: newest.next = q.next, q.next = newest. With only one pointer, you'd have lost the reference to the insertion predecessor.
Q23INTERVIEWHow would you merge two sorted linked lists into one sorted list?
Answer — Two Pointer Merge
Use a dummy head node. Compare the current heads of both lists; append the smaller one to the result, advance that pointer. Append remaining elements when one list is exhausted.
Approach: Node dummy = new Node(0, null);
Node curr = dummy;
while (a != null && b != null) {
  if (a.element <= b.element) { curr.next = a; a = a.next; }
  else { curr.next = b; b = b.next; }
  curr = curr.next;
}
curr.next = (a != null) ? a : b;
return dummy.next;
Time: O(n+m) · Space: O(1)

Q25INTERVIEWC#'s LinkedList<T> is doubly-linked and supports First/Last direct access. Name two data structures that are commonly implemented using it.
Answer
  • Deque (Double-Ended Queue) — AddFirst/AddLast/RemoveFirst/RemoveLast are all O(1). Perfect for deque semantics.

Q26INTERVIEWWhat is an LRU Cache and how does a linked list help implement it efficiently?
Answer
An LRU (Least Recently Used) Cache keeps the N most recently accessed items and evicts the least recently used when full.

Implementation using LinkedList<T> + Dictionary<K, LinkedListNode>:
  • Access/Add: Move node to front (Remove(node) = O(1), AddFirst = O(1)) — total O(1)
  • Evict: RemoveLast() = O(1) — removes LRU item
  • Lookup: Dictionary maps key → node reference for O(1) access
Without a doubly-linked list, moving a node to front would be O(n). The doubly-linked structure makes the entire cache O(1) per operation.

Q28GOTCHACan you pass a node from one LinkedList<T> to AddBefore() of a different LinkedList<T>? What happens?
Answer
No. LinkedListNode<T> has a List property that records which list it belongs to. If you pass a node that belongs to a different list (or has already been removed/cleared), AddBefore(), AddAfter(), and Remove(node) all throw InvalidOperationException. Always use nodes from the same list instance.
Q29WHYWhy is iteration over a linked list slower than over an array in practice, even though both are O(n)?
Answer — Cache Locality
Arrays store elements contiguously — when the CPU loads one element, it prefetches the next cache line (64 bytes = 16 ints) automatically. Almost every access is a cache hit. Linked list nodes are scattered across RAM — each node.next pointer dereference likely causes a cache miss, requiring a fresh RAM fetch (~100ns vs ~1ns for cache). For n=1,000,000 elements, iterating a linked list can be 5–10× slower than an array despite the same O(n) Big-O.
Q30INTERVIEWHow do you check if a linked list is a palindrome?
Answer
Three steps: (1) Find the middle using slow/fast pointers. (2) Reverse the second half in-place. (3) Compare first half and reversed second half node-by-node.
Result: If all values match → palindrome. Optionally restore the list by reversing the second half again. Time: O(n) · Space: O(1).
Q31INTERVIEWExplain the Count property vs manually counting nodes with a loop. When does the difference matter?
Answer
Count (built-in) and size (custom) are maintained as an integer field — accessing them is O(1). Counting nodes with a traversal loop is O(n). The difference matters when you call Count (or length()) inside a loop:
for (int i = 0; i < list.Count; i++) ... — this is O(n) total since Count is O(1).
If Count were O(n), the loop would be O(n²). Always prefer stored-counter implementations for size queries on production linked lists.
