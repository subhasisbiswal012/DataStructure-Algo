🔍 Linear Search — Sequential Scan

💡 Linear search is the simplest search. Start at index 0, check each element one by one. If it matches the key, return the index. If you reach the end without a match, return −1. No sorting required.

Algorithm Steps

  1. Set i = 0
  2. While i < n: compare arr[i] with key
  3. If match → return i
  4. Else → i++, repeat
  5. If loop ends → return -1
csharp
public int LinearSearch(int[] arr, int key)
{
    int i = 0;
    while (i < arr.Length)
    {
        if (arr[i] == key)
            return i;     // ✅ found — return index
        i++;              // move to next
    }
    return -1;            // ❌ not found
}

⚡ Binary Search (Iterative) — Divide & Conquer

💡 Binary search works on a sorted array. Instead of checking every element, it looks at the middle element and eliminates half the remaining search space each time. After k steps, only n / 2ᵏ elements remain.
⚠️ Prerequisite: the array MUST be sorted in ascending order. Binary search on an unsorted array gives wrong results — no error, just incorrect output.

Algorithm Steps

  1. Set L = 0, R = n-1
  2. While L ≤ R:
    • Compute M = ⌊(L + R) / 2⌋
    • If arr[M] == key → return M
    • If key < arr[M] → R = M - 1 (search left half)
    • If key > arr[M] → L = M + 1 (search right half)
  3. If loop ends without match → return -1
csharp
public int BinarySearch(int[] arr, int key)
{
    int L = 0;
    int R = arr.Length - 1;

    while (L <= R)
    {
        int M = (L + R) / 2;   // middle index

        if (arr[M] == key)
            return M;           // ✅ found
        else if (key < arr[M])
            R = M - 1;          // search LEFT half
        else
            L = M + 1;          // search RIGHT half
    }
    return -1;                  // ❌ not found
}

🔄 Binary Search (Recursive)

💡 Same logic as iterative binary search — but instead of a while loop, each comparison triggers a recursive call with updated L or R bounds. The base case is when L > R (nothing left to search).
csharp
public int BinarySearchRec(int[] arr, int key, int L, int R)
{
    if (L > R)
        return -1;           // base case: not found

    int M = (L + R) / 2;

    if (arr[M] == key)
        return M;            // ✅ found
    else if (key < arr[M])
        return BinarySearchRec(arr, key, L, M - 1);   // LEFT
    else
        return BinarySearchRec(arr, key, M + 1, R);   // RIGHT
}

// Call: BinarySearchRec(arr, key, 0, arr.Length - 1)

Q01DEF What is linear search and what does it return when the key is not found?
Answer
Linear search (also called sequential search) scans each element of the array from index 0 onwards, comparing each with the key. If a match is found, it returns the index. If the entire array is traversed with no match, it returns −1.
Example: Searching for 100 in [84, 21, 47, 96, 15] → checks 84, 21, 47, 96, 15 → no match → returns −1.
Q02APPLY Write the C# while-loop implementation of linear search. What are its three parameters?
Answer
csharp
public int LinearSearch(int[] arr, int key)
{
    int i = 0;
    while (i < arr.Length)
    {
        if (arr[i] == key) return i;
        i++;
    }
    return -1;
}
Parameters: arr (the array), key (element to find). arr.Length is the size.
Q03WHY Why is linear search's worst-case O(n)? Walk through the operation count.
Answer
  • Init i = 0: O(1)
  • While condition i < n: executed n+1 times
  • If comparison: executed n times (worst case — no match)
  • Increment i++: n times
  • Return −1: O(1)
Total ≈ 3n + 3 → O(n). In the worst case (key not found or at the end), every element is visited.
Q04DEF What is the critical prerequisite for binary search? What happens if it's violated?
Answer
The array must be sorted in ascending (or descending) order. If binary search is applied to an unsorted array, it will give wrong results — it won't crash, it will simply return an incorrect index or −1 even when the element exists.
Why: Binary search eliminates half the array based on whether key is less or greater than mid. This logic only works if elements are ordered.
Q05SEQUENCE Trace binary search for key = 25 in [10, 20, 30, 40, 50]. Show every value of L, M, R.
Answer
  • Step 1: L=0, R=4, M=2, arr[2]=30. 25 < 30 → R = M-1 = 1
  • Step 2: L=0, R=1, M=0, arr[0]=10. 25 > 10 → L = M+1 = 1
  • Step 3: L=1, R=1, M=1, arr[1]=20. 25 > 20 → L = M+1 = 2
  • Stop: L=2 > R=1 → return −1 (not found)
Q06APPLY Write the iterative binary search in C#. What is the loop condition and how is M calculated?
Answer
csharp
public int BinarySearch(int[] arr, int key)
{
    int L = 0, R = arr.Length - 1;
    while (L <= R)
    {
        int M = (L + R) / 2;    // floor division
        if (arr[M] == key) return M;
        else if (key < arr[M]) R = M - 1;
        else L = M + 1;
    }
    return -1;
}
Loop condition: L ≤ R. M = (L + R) / 2 (integer division = floor). The floor is automatic in C# integer arithmetic.
Q07WHY Why is binary search O(log n)? Explain using the halving principle.
Answer
After each iteration, the search space is halved: n → n/2 → n/4 → … → 1. The number of times you can halve n before reaching 1 is log₂n. So the loop runs at most log₂n times. Each iteration does O(1) work → total is O(log n).
Numbers: n = 1,000,000 → log₂(1,000,000) ≈ 20 iterations. Linear search would take up to 1,000,000.
Q08GOTCHA When does binary search update R = M - 1 vs L = M + 1? What happens if you forget the ±1?
Answer
  • R = M - 1: when key < arr[M] — key is in the LEFT half. Exclude M itself.
  • L = M + 1: when key > arr[M] — key is in the RIGHT half. Exclude M itself.
Forgetting the ±1 (writing R = M or L = M) causes an infinite loop — the search space never shrinks when L = R = M.
Q09APPLY Write the recursive binary search in C#. What is the base case?
Answer
csharp
public int BinarySearchRec(int[] arr, int key, int L, int R)
{
    if (L > R) return -1;       // base case: empty range
    int M = (L + R) / 2;
    if (arr[M] == key) return M;
    else if (key < arr[M])
        return BinarySearchRec(arr, key, L, M - 1);
    else
        return BinarySearchRec(arr, key, M + 1, R);
}
Base case: L > R — the search range is empty, so the key cannot exist → return −1.
Q10COMPARE Compare iterative vs recursive binary search on time and space complexity. Which is preferred in production?
Answer
  • Both have O(log n) time complexity
  • Iterative space: O(1) — only L, R, M variables, no stack frames
  • Recursive space: O(log n) — each call pushes a stack frame; max depth = log n
Iterative is preferred in production for large arrays because recursive can stack overflow and uses more memory. Recursive is cleaner to read and reason about.
Q11SEQUENCE Trace recursive binary search for key = 50 in [10, 20, 30, 40, 50, 60, 70]. Show every recursive call.
Answer
  • Call 1: BS(0,6) → M=3, arr[3]=40. 50 > 40 → call BS(4,6)
  • Call 2: BS(4,6) → M=5, arr[5]=60. 50 < 60 → call BS(4,4)
  • Call 3: BS(4,4) → M=4, arr[4]=50. 50 == 50 → return 4 ✅
Call stack unwinds: Call 3 returns 4 → Call 2 returns 4 → Call 1 returns 4.
Q12GOTCHA You run binary search on [5, 2, 8, 1, 9] looking for 8. It returns −1. Why?
Answer
The array is not sorted. Binary search requires sorted input. On [5,2,8,1,9]: L=0, R=4, M=2, arr[2]=8 — ironically it would find 8 here, but this is luck. Change the target to 1: L=0, R=4, M=2, arr[2]=8, 1<8 → R=1. Then L=0, R=1, M=0, arr[0]=5, 1<5 → R=-1. Stop → returns -1. Wrong! 1 is at index 3. Always sort before binary searching.
Q13WHY Why does the recursive binary search have O(log n) space complexity?
Answer
Each recursive call adds a stack frame holding L, R, M and the return address. Because each call halves the search space, the maximum recursion depth is log₂n. Therefore at most log n frames exist simultaneously on the call stack → O(log n) space.
Example: n = 1,000,000 → max stack depth ≈ 20 frames. Negligible in practice, but not O(1).
Q14COMPARE Linear search vs binary search on a sorted array of 1 million elements. Which is better and why?
Answer
Binary search is vastly better. Linear search does up to 1,000,000 comparisons. Binary search does at most log₂(1,000,000) ≈ 20 comparisons. That's ~50,000× faster. For large, already-sorted data, binary search is the clear choice.
Q15DEF What do L, R, and M represent in binary search, and what formula computes M?
Answer
  • L (Left) — the start index of the current active search range
  • R (Right) — the end index of the current active search range
  • M (Mid) — the middle index; element being compared against the key
Formula: M = ⌊(L + R) / 2⌋. Integer division in C# gives the floor automatically.
Q16GOTCHA What is the termination condition for the iterative binary search loop, and why is it L ≤ R not L < R?
Answer
The condition is L ≤ R because when L = R, there is still one element left to check. Using L < R would skip this case — if the key is at that final single-element position, the function would incorrectly return −1 even though the element exists.
Example: Searching for 10 in [10]. L=0, R=0. L≤R → M=0, arr[0]=10 → found. If using L<R: loop never runs → wrong return of −1.
Q17APPLY For binary search on [10, 20, 30, 40, 50, 60, 70] searching for 40: what does L, M, R equal in the very first iteration?
Answer
L = 0, R = 6, M = ⌊(0+6)/2⌋ = 3. arr[3] = 40 = key → found immediately, return 3. This is the best case: O(1) — the key is exactly at the midpoint on the first comparison.
Q18WHY When would you prefer linear search over binary search even if the array is sorted?
Answer
  • Very small arrays (n < ~8–16): overhead of binary search setup isn't worth it; linear is simpler
  • Single search on data that needs sorting first: sorting costs O(n log n) upfront; if you only search once, linear O(n) is cheaper overall
  • Linked lists: can't do random access to compute mid efficiently — linear is natural
Q19SEQUENCE List the three decision branches inside binary search and what each does to L and R.
Answer
  1. arr[M] == key → return M (found!)
  2. key < arr[M] → R = M − 1 (discard right half, search left)
  3. key > arr[M] → L = M + 1 (discard left half, search right)

Q21GOTCHA What is the recurrence relation for recursive binary search and what is its solution?
Answer
Recurrence: T(n) = T(n/2) + O(1), T(1) = O(1)

Each call does O(1) work (compute M, one comparison) and makes one recursive call on half the array.

Solution: T(n) = T(n/2) + 1 = T(n/4) + 1 + 1 = … = T(1) + log₂n = O(log n)
