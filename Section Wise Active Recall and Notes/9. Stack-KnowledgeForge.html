Stack Data Structure

1. What is a Stack?

💡 Think of a stack of plates in a cafeteria. You can only add a plate to the top, and you can only take a plate from the top. The last plate placed is the first one removed.

A stack is a linear data structure that follows the LIFO principle — Last In, First Out. Whatever was inserted most recently is the first thing that comes out. You can insert at any time, but you can only remove from one end: the top.

  • Push — insert an element onto the top
  • Pop — remove and return the top element
  • Top / Peek — see the top element without removing it
  • IsEmpty — check if the stack has no elements
  • Length / Count — number of elements currently in the stack
📌 The only access point is the top. There is no "insert at position 3" or "read the bottom element" in a true stack ADT.

2. Real-World Applications

💡 Stacks appear everywhere the phrase "undo the last action" or "go back to where you were" makes sense.
⚙️
Function Call Stack
Every function call is pushed; when it returns, it's popped. Stack overflow = running out of stack space.
↩️
Undo / Redo
Ctrl+Z in editors. Each action is pushed; undo pops the last action and reverses it.
🌐
Browser Back Button
Every page you visit is pushed. Pressing Back pops and returns to the previous URL.
🏷️
HTML Tag Validation
Parser pushes opening tags, pops when closing tag is found. Mismatched = syntax error.
🔢
Expression Evaluation
Evaluating 3 + 4 * 2 uses a stack to handle operator precedence correctly.
🔄
Infix → Postfix
Shunting-yard algorithm uses a stack to convert a+b*c to abc*+.
🧩
Parentheses Matching
Push (, [, {; pop and verify match when closing bracket appears.
🗂️
DFS / Graph Traversal
Depth-First Search uses a stack (explicit or via recursion) to track which path to explore next.
💾
Memory Management
The call stack in every runtime (JVM, CLR, V8) is literally a stack — frames pushed and popped per call.
🖨️
Printer Spooler (LIFO mode)
Some print queues prioritise the most recent document — last sent, first printed.

3. C# Built-in Stack<T>

💡 C# provides a generic Stack<T> class in System.Collections.Generic. It's ready to use — no implementation needed.
csharp
using System.Collections.Generic;

Stack<int> stack = new Stack<int>();

stack.Push(10);      // push 10
stack.Push(20);      // push 20
stack.Push(30);      // push 30

int top   = stack.Peek();   // 30 — does NOT remove
int popped = stack.Pop();   // 30 — REMOVES and returns
bool empty = stack.IsEmpty(); // false — 2 elements remain (note: extension method or Count==0)
int count = stack.Count;    // 2

stack.Contains(10);  // true — searches entire stack
stack.Clear();       // empties the stack

4. Internals — Dynamic Array Under the Hood

💡 C#'s Stack<T> is internally backed by a dynamic array (just like List<T>). Understanding why requires understanding what a dynamic array is.

What is a Dynamic Array?

A dynamic array starts with a fixed-size internal buffer. When it runs out of space, it:

  1. Allocates a new, larger array (typically 2× the current size)
  2. Copies all existing elements to the new array
  3. Releases the old array (garbage collected)
  4. Continues accepting new elements

Why a Dynamic Array (not a fixed array)?

  • Unknown size at declaration time — you don't know how many items you'll push before the program runs
  • No wasted pre-allocation — starts small, grows only as needed
  • O(1) top access — the "top" is always the last index of the array, reachable instantly
  • Cache friendly — array elements are contiguous in memory, so modern CPU caches prefetch them efficiently
📌 Doubling strategy is key. If you grew by 1 each time, every push would be O(n). Doubling ensures that resizes become exponentially rare, giving O(1) amortised.

5. Array-Based vs Linked List-Based Stack

💡 Both implement the same Stack ADT. The difference is what sits underneath — and that affects memory, speed, and behaviour under load.
Array-Based Stack
  • Push/Pop: O(1) amortised (occasional O(n) resize)
  • Pre-allocates capacity — some wasted space possible
  • Cache-friendly — elements are contiguous
  • Resizing copies entire array — brief O(n) spike
  • Fixed overhead: just an array + top index
  • No pointer overhead per element
Linked List-Based Stack
  • Push/Pop: O(1) always — no resize ever
  • No wasted capacity — allocates exactly one node per push
  • Each node has pointer overhead (8 bytes on 64-bit)
  • Nodes scattered in memory — CPU cache misses more likely
  • Never has a sudden O(n) spike
  • Better for real-time systems where predictable latency matters

Linked List Stack — C# Implementation

csharp
public class Node<T>
{
    public T element;
    public Node<T> next;
    public Node(T e, Node<T> n) { element = e; next = n; }
}

public class LinkedStack<T>
{
    private Node<T> _top = null;
    private int _size = 0;

    public int Count    => _size;
    public bool IsEmpty => _size == 0;

    public void Push(T item)
    {
        _top = new Node<T>(item, _top);   // new node points to old top; head moves
        _size++;
    }

    public T Pop()
    {
        if (IsEmpty) throw new InvalidOperationException("Stack is empty.");
        T val = _top.element;
        _top  = _top.next;                 // advance top; old top becomes unreferenced
        _size--;
        return val;
    }

    public T Peek()
    {
        if (IsEmpty) throw new InvalidOperationException("Stack is empty.");
        return _top.element;
    }
}
📌 In a linked-list stack, head = top of stack. Push = addFirst. Pop = removeFirst. Both are strict O(1) — no resize possible.

6. Classic Stack Algorithms with Code

Balanced Parentheses Checker

csharp
public static bool IsBalanced(string expression)
{
    var stack = new Stack<char>();
    foreach (char c in expression)
    {
        if (c == '(' || c == '[' || c == '{')
            stack.Push(c);
        else if (c == ')' || c == ']' || c == '}')
        {
            if (stack.Count == 0) return false;    // closing with nothing open
            char top = stack.Pop();
            if ((c == ')' && top != '(') ||
                (c == ']' && top != '[') ||
                (c == '}' && top != '{'))
                return false;                       // mismatched pair
        }
    }
    return stack.Count == 0;   // true only if all opened tags were closed
}
// IsBalanced("({[]})") → true
// IsBalanced("({[})") → false

Q01DEF What is a stack? Define it formally and state its core property.
Answer
A stack is a linear data structure that follows the LIFO (Last In, First Out) principle — the most recently inserted element is always the first to be removed. It supports insertion and removal from only one end, called the top. Elements at the bottom are inaccessible until everything above them is popped.
Analogy: A stack of plates — you can only add or remove from the top.

Q03WHY Why is Push O(1) amortised rather than strictly O(1) in an array-based stack?
Answer
The internal array has a fixed capacity. When it fills up, Push triggers a resize: a new array of double the size is allocated, and all n existing elements are copied. This single push costs O(n). However, because the array doubles each time, resizes become exponentially rare — the average cost per push across n operations works out to O(1). That's why we say amortised O(1).
Mental model: After 8 pushes at capacity 8, you pay O(8) for the resize. Spread over 8 pushes, each effectively cost O(1).
Q04COMPARE Compare array-based and linked list-based stack implementations. When would you prefer each?
Answer
Array-based: Cache-friendly (contiguous memory), less memory per element, but occasional O(n) resize spike. Use when: memory efficiency and average-case throughput matter (most general cases — C#'s Stack<T> is this).

Linked list-based: Strictly O(1) push/pop always — no resize. Each node has pointer overhead and is scattered in memory (cache unfriendly). Use when: predictable worst-case latency is critical (real-time systems, embedded).
Q05GOTCHA What exception does C#'s Stack<T>.Pop() or Peek() throw on an empty stack, and how do you avoid it?
Answer
Both throw InvalidOperationException when called on an empty stack. Avoid it by either checking stack.Count > 0 before calling, or using the safe variants: TryPop(out T result) and TryPeek(out T result) — these return false instead of throwing.
Best practice: In production code, always use TryPop/TryPeek. The regular Pop/Peek is fine only when you've already verified Count > 0.
Q06APPLY Using only a stack, reverse a string "hello". Walk through the steps.
Answer
1. Push each character: push 'h', 'e', 'l', 'l', 'o' → stack top = 'o'
2. Pop each character and append: pop 'o', 'l', 'l', 'e', 'h' → "olleh"

csharp
string Reverse(string s)
{
    var stack = new Stack<char>();
    foreach (char c in s) stack.Push(c);
    var sb = new System.Text.StringBuilder();
    while (stack.Count > 0) sb.Append(stack.Pop());
    return sb.ToString();
}
Q07WHY Why is a stack used for function call management (the call stack)? Why not a queue?
Answer
When function A calls function B, B must finish first before A can continue. This is exactly LIFO — the most recently called function is always the one that returns next. A queue would require FIFO — meaning A would return before B, which makes no sense for nested calls. The call stack models the natural nesting of function calls.
Extension: Stack overflow = the call stack grows beyond its memory limit, usually caused by infinite recursion.
Q08SEQUENCE Push 5, 10, 15 onto a stack, then call Pop() twice and Peek() once. What is returned in order?
Answer
After push(5), push(10), push(15) → stack = [5, 10, 15] (15 on top)
Pop() #1 → returns 15. Stack = [5, 10]
Pop() #2 → returns 10. Stack = [5]
Peek() → returns 5. Stack still = [5] (not removed)
Q09APPLY Implement a MinStack — a stack that supports Push, Pop, and GetMin() in O(1) time.
Answer
Use two stacks — one for data, one to track the current minimum at each level.
csharp
public class MinStack
{
    private Stack<int> _data = new();
    private Stack<int> _mins = new();

    public void Push(int val)
    {
        _data.Push(val);
        int currentMin = _mins.Count == 0 ? val : Math.Min(val, _mins.Peek());
        _mins.Push(currentMin);
    }
    public int Pop()    { _mins.Pop(); return _data.Pop(); }
    public int Peek()   => _data.Peek();
    public int GetMin() => _mins.Peek();   // O(1)!
}
Key: _mins tracks the minimum at each stack level. When you pop from _data, pop the corresponding min too.
Q10GOTCHA What is "stack overflow" in the context of programming runtimes? What causes it?
Answer
Stack overflow occurs when the call stack (the runtime's stack of function frames) exceeds its allocated memory limit. Each function call pushes a frame; if frames are pushed faster than they're popped (e.g., infinite recursion with no base case), the stack runs out of space. The runtime throws a StackOverflowException in .NET.
Example: void Recurse() { Recurse(); } — no base case, infinite push, stack overflow.
Q11APPLY Implement a queue using two stacks. What is the time complexity of enqueue and dequeue?
Answer
csharp
public class QueueUsingStacks<T>
{
    private Stack<T> _inbox  = new();  // for enqueue
    private Stack<T> _outbox = new();  // for dequeue

    public void Enqueue(T item) => _inbox.Push(item);  // O(1)

    public T Dequeue()
    {
        if (_outbox.Count == 0)
            while (_inbox.Count > 0)
                _outbox.Push(_inbox.Pop());   // transfer reverses order → FIFO
        if (_outbox.Count == 0) throw new InvalidOperationException();
        return _outbox.Pop();
    }
}
Complexity: Enqueue = O(1). Dequeue = O(1) amortised (each element transferred at most once across all dequeues).
Q12DEF What is a dynamic array? Why does C#'s Stack<T> use one instead of a fixed-size array?
Answer
A dynamic array is an array that automatically grows when its capacity is exceeded — it allocates a new larger buffer (typically 2× size), copies all elements, and discards the old buffer. C# uses it in Stack<T> because: (1) you don't know how many elements will be pushed at compile time, (2) a fixed array would either waste space or run out, (3) the dynamic approach gives O(1) amortised push with no upfront over-allocation.

Q14WHY Why is the linked list-based stack preferred in real-time systems over the array-based stack?
Answer
Real-time systems require predictable, bounded latency. The array-based stack has an occasional O(n) resize that can take an unpredictable amount of time — in a real-time system (medical device firmware, flight control, trading systems), a sudden O(n) spike at the wrong moment is unacceptable. The linked list stack guarantees strictly O(1) push and pop — no resize, no spikes, no surprises.
Q15GOTCHA Does foreach on a C# Stack<T> iterate from top to bottom or bottom to top?
Answer
Top to bottom. The stack's enumerator starts at the most recently pushed element (top) and works toward the oldest (bottom). If you push 1, 2, 3, foreach yields 3, 2, 1. This is the opposite of the insertion order — a common source of bugs when people assume bottom-to-top.
Gotcha: foreach does NOT pop the elements — the stack remains intact after iteration.
Q16COMPARE What is the difference between a stack and a queue? Give a real-world example of each.
Answer
Stack — LIFO: Last element in is the first element out. Access only from the top.
Real-world: undo/redo in a text editor — the last action is the first to be undone.

Queue — FIFO: First element in is the first element out. Insert at rear, remove from front.
Real-world: a ticket queue at a cinema — first person in line is the first served.
Q17APPLY How would you use a stack to evaluate the postfix expression "5 3 + 2 *"?
Answer
Scan left to right:
See 5 → push 5. Stack: [5]
See 3 → push 3. Stack: [5, 3]
See + → pop 3 and 5, compute 5+3=8, push 8. Stack: [8]
See 2 → push 2. Stack: [8, 2]
See * → pop 2 and 8, compute 8*2=16, push 16. Stack: [16]
End → result = 16
Rule: Operand → push. Operator → pop two, apply, push result.
Q18SEQUENCE What is the space complexity of the C# built-in Stack<T>? What contributes to it?
Answer
Overall space: O(n) where n is the number of elements pushed. Contributions: (1) the internal array holding n elements of type T, (2) the capacity — which may be up to 2n after a resize (empty slots), (3) a few O(1) tracking variables (count, version). The extra empty capacity is bounded by a constant factor of n, so overall is still O(n).
Q19GOTCHA Can you iterate a Stack<T> while modifying it (push/pop during foreach)? What happens?
Answer
No. C#'s Stack<T> (like most .NET collections) tracks a version counter that increments on every structural modification. The enumerator checks this on every MoveNext() call. If the version has changed, it throws InvalidOperationException: Collection was modified; enumeration operation may not execute.
Fix: Snapshot the stack first (var copy = new Stack<T>(stack.ToArray())) and iterate the copy if you need to modify the original.
Q20COMPARE What is the difference between Stack<T> in System.Collections.Generic and Stack in System.Collections (non-generic)? Which should you use and why?
Answer
System.Collections.Stack (legacy): Non-generic — stores object. Any type can be pushed, but every read requires a cast. Boxing/unboxing happens for value types (int, bool, etc.) which is slower and generates garbage. No compile-time type safety.

System.Collections.Generic.Stack<T> (modern): Generic — type-safe at compile time. No boxing for value types. Better performance and catches type errors at compile time instead of runtime.

Always use Stack<T> unless maintaining legacy .NET Framework code.
