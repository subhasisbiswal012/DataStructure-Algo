🚦 1. Priority Queues

💡 Think of a hospital emergency room — patients don't get treated in arrival order. The most critical case goes first. That's a priority queue.

What is a Priority Queue?

A Priority Queue is an abstract data type where each element has an associated priority key. Unlike a regular queue (FIFO), elements are removed in priority order — the highest priority element is always served first, regardless of when it was inserted.

  • Elements are inserted just like a regular queue.
  • Elements are removed based on priority (smallest key = highest priority in min-heap; largest key = highest priority in max-heap).
  • Priority is a separate value assigned at insertion time.

🌳 3. Heap Data Structure — Definition & Properties

💡 A Heap is the most efficient way to implement a priority queue. Think of it as a partially sorted binary tree — not fully sorted, but with one guarantee: every parent is "better" than its children.

What is a Heap?

A Heap (also called a Binary Heap) is a binary tree that satisfies two strict properties simultaneously:

🔵 Structural Property
Must be a Complete Binary Tree:
• Every level is fully filled except possibly the last level
• Last level is filled from left to right, no gaps
🟢 Relational Property
For a Max Heap: every parent ≥ all its children
For a Min Heap: every parent ≤ all its children
Rule applies at every parent-child pair in the tree
⚠️ A Heap is NOT a Binary Search Tree (BST). BST requires left child < parent < right child. A heap only cares about parent vs children — sibling nodes have no required ordering.

Array Representation — The Key Insight

Heaps are stored in arrays (not actual tree nodes) using a mathematical index relationship:

  • Root is stored at index 1 (index 0 is unused for clean math)
  • For node at index i: Left child = 2i, Right child = 2i+1
  • For node at index i: Parent = ⌊i/2⌋ (integer division)

⬆️ 7. Insertion — Concept & Graphical Representation

📌 Two-Phase Insert: (1) Place the new node at the next available position to maintain the complete binary tree shape. (2) Bubble it UP by swapping with its parent until the relational property is satisfied.

Insert Process Step-by-Step

  1. Place new node at the next available slot (leftmost open spot in the last level, or start of next level if full).
  2. Compare it with its parent: new > parent in a max heap?
  3. If yes → swap (bubble up). If no → done.
  4. Repeat from the new position until reaching root or no swap needed.

Insertion Sequence: 20 → 14 → 2 → 15 → 10 → 21

Insert O(1) — placement O(log n) — bubbling worst case O(1) space

⚙️ 8. Insert — Algorithm & Code Walkthrough

💡 The algorithm uses a single variable trick: instead of swapping twice per step, it bubbles the parent value DOWN and saves the new value until its final position is found. Then it assigns once.
csharp — Insert()
public void Insert(int value)
{
    // Guard: heap full
    if (currentSize == maxSize)
    {
        Console.WriteLine("Heap is full.");
        return;
    }

    // ① Place at next slot
    currentSize++;
    int index = currentSize;

    // ② Bubble UP while:
    //    - not at root (index > 1)
    //    - value greater than parent
    while (index > 1 &&
           value > data[index / 2])
    {
        // Move parent DOWN
        data[index] = data[index / 2];

        // Jump UP to parent position
        index /= 2;
    }

    // ③ Final placement
    data[index] = value;
}
Trace: Insert 21 into [20, 15, 2, 14, 10]
① Place at index 6
currentSize++ → 6. index=6. Array[6]=placeholder. value=21.
② Loop iter 1: index=6, parent=index/2=3
data[3]=2. value(21) > data[3](2) → move 2 down: data[6]=2. index=3.
② Loop iter 2: index=3, parent=index/2=1
data[1]=20. value(21) > data[1](20) → move 20 down: data[3]=20. index=1.
② Loop exits: index=1 (root)
index > 1 is false. Exit loop.
③ Place: data[1] = 21
Result: [_,21,15,20,14,10,2] ✓
⚠️ Key detail in the code: The loop condition is j < currentSize — NOT j <= currentSize. This matters for the last element: if j == currentSize, there's only a left child, no right. Accessing data[j+1] would be out of bounds without the right condition.

⬇️ 9. Deletion — Concept & 3 Tree Examples

📌 Always delete the ROOT (max element in max heap). Replace it with the last node, remove that last node, then bubble DOWN to restore the relational property.

Why replace with the last node?

  • Simply removing the root leaves two disconnected subtrees — structural property broken.
  • Moving the last node to root preserves the complete binary tree shape (just reduces size by 1).
  • The last node will likely violate the relational property at root — fix it by bubbling DOWN.

Down-Heap Bubbling Rules

  • Compare the displaced node with its larger child (in max heap).
  • If the node is smaller than the larger child → swap.
  • Continue downward until no swap needed or reached a leaf.
  • Edge case: if node has only a left child, compare only with that one.

🗑️ 10. Delete — Algorithm & Code Walkthrough

csharp — Delete()
public int Delete()
{
    if (IsEmpty)
    {
        Console.WriteLine("Heap empty.");
        return -1;
    }

    // ① Save root — this is our return value
    int deletedRoot = data[1];

    // ② Move last node to root
    data[1] = data[currentSize];
    data[currentSize] = -1; // clear last slot
    currentSize--;

    // ③ Down-heap bubble
    int i = 1;          // start at root
    int j = 2 * i;      // left child

    while (j <= currentSize)
    {
        // Pick larger child (if right exists)
        if (j < currentSize &&
            data[j] < data[j + 1])
        {
            j++; // right child is larger
        }

        // Swap if parent < larger child
        if (data[i] < data[j])
        {
            int temp = data[i];
            data[i] = data[j];
            data[j] = temp;

            // Move down
            i = j;
            j = 2 * i;
        }
        else
        {
            break; // heap property satisfied
        }
    }

    return deletedRoot;
}
Delete 21 from [21, 15, 20, 14, 10, 2]
① Save root: deletedRoot = 21
data[1] = 21. We'll return this at the end.
② Move last to root
data[1] = data[6] = 2. data[6] = -1. currentSize = 5.
Array: [_, 2, 15, 20, 14, 10]
③ Loop i=1, j=2
j < currentSize(5): compare j(15) vs j+1(20). 15 < 20 → j=3 (right is bigger). data[i](2) < data[j](20) → SWAP. i=3, j=6.
③ Loop exits: j=6 > currentSize(5)
Array: [_, 20, 15, 2, 14, 10]. Done.
Return 21 ✓
Heap is valid. Root is now 20 (next maximum).
⚠️ Critical: condition is j < currentSize when checking right child — not j+1 <= currentSize. Same thing but easier to miss.

🔀 11. Heap Sort — Concept

💡 Heap Sort is essentially: "Build a max-heap, then keep deleting the max. Each deleted max goes to the end of the array — building sorted order from back to front."

Two Phases

  • Phase 1 — Build Heap: Insert all n elements from the array into a max-heap, one by one. Each insert is O(log n). Total: O(n log n).
  • Phase 2 — Extract Sorted: Delete the root (max) n times. Each delete is O(log n). Write deleted elements back into the array from index n-1 down to 0. Total: O(n log n).
📌 Deletion always gives the current maximum. Writing it from right to left fills the array in ascending order. Final array: [smallest ... largest].

🔢 12. Heap Sort — Algorithm & Code Walkthrough

csharp — HeapSort()
public void HeapSort(int[] arr)
{
    int n = arr.Length;
    Heap heap = new Heap();

    // ── Phase 1: Build heap ──────────────────
    // Insert every element from arr into heap
    // Each insert: O(log n)
    // Total phase 1: O(n log n)
    for (int i = 0; i < n; i++)
    {
        heap.Insert(arr[i]);
    }

    // ── Phase 2: Extract sorted ──────────────
    // Delete root (max) n times
    // Write each deleted max from arr[n-1] downward
    // Each delete: O(log n)
    // Total phase 2: O(n log n)
    for (int i = n - 1; i >= 0; i--)
    {
        arr[i] = heap.Delete();
    }
    // arr now contains elements in ascending order
}

// Usage:
static void Main(string[] args)
{
    int[] ints = { 12, 345, 13, 4325, 123, 4525 };
    new Heap().HeapSort(ints);
    // ints = [12, 13, 123, 345, 4325, 4525]
    foreach (int num in ints)
        Console.Write(num + " ");
}

📚 Course Notes — Fundamentals (Q1–Q10)
Q01DEFWhat is a Priority Queue and how does it differ from a regular Queue?
Answer
A Priority Queue is a collection where each element has an assigned priority key. Elements are removed in priority order — the highest priority element exits first, regardless of insertion order. A regular queue is strict FIFO (first in, first out): the element that was inserted earliest always exits first. Priority queue breaks FIFO — a late-arriving element with higher priority can exit before earlier ones.
Q02DEFWhat are the two properties every valid heap must satisfy? Describe each.
Answer
1. Structural Property: The heap must be a complete binary tree — every level except possibly the last must be fully filled, and the last level must be filled left-to-right with no gaps.

2. Relational Property: For a max heap, every parent node's key must be ≥ the keys of all its children. For a min heap, every parent must be ≤ all its children. This rule applies at every parent-child pair in the entire tree.
Q03DEFWhat is the parent-child index formula for a heap stored in an array? Why is index 0 left unused?
Answer
For a node at index i:
  • Left child = 2i
  • Right child = 2i + 1
  • Parent = ⌊i / 2⌋ (integer division)
Index 0 is left unused because if root were at index 0: parent of node at index i would be ⌊(i-1)/2⌋ — messier formula, and left child of 0 would be at index 1 but right child at index 2 doesn't fit cleanly. Starting at index 1 keeps the math clean: parent = i/2 always works with no edge cases.
Q04WHYWhy is a Heap NOT a Binary Search Tree (BST)? What's the key difference?
Answer
A BST requires: left child < parent < right child — a strict ordering relationship between siblings and all descendants. In a heap, siblings have no required ordering relative to each other. The only rule is parent vs its direct children. So in a max heap, the left child could be 15 and right child could be 20 — both are less than parent (21), which is valid. But in a BST, left must always be less than right's subtree, which isn't required here. Heap = partial ordering. BST = total ordering.
Q05APPLYWhat does "up-heap bubbling" mean during insertion, and when does it stop?
Answer
After placing a new element at the next available slot (maintaining structural property), up-heap bubbling restores the relational property. It works like this: compare the new element with its parent. If new > parent (max heap) → swap them. Then compare with the new parent. Repeat.

It stops when either: (1) the element is no longer greater than its parent — relational property satisfied, or (2) the element reaches the root (index 1) — can't go higher.

Q07WHYWhy does the Insert algorithm move the parent value DOWN instead of doing a traditional two-variable swap?
Answer
A traditional swap requires a temp variable and 3 assignments per level. The algorithm's approach is more efficient: it holds the new value in a variable, then shifts parents down without writing the new value at each level. The new value is only written once, at its final position. This cuts the total write operations by roughly half. It's also cleaner in the loop — only one move per iteration instead of three.
Q08APPLYDuring deletion, why do we replace the root with the LAST node specifically, and not any other node?
Answer
Replacing with the last node is the only choice that preserves the structural property (complete binary tree). If we removed the root and left the two subtrees, we'd have a disconnected tree — no longer a valid structure. If we replaced with a random middle node, we'd create a gap, violating the complete binary tree requirement. The last node is the only node whose removal doesn't create a gap. After moving it to root, we restore the relational property through down-bubbling.

Q10WHYWhy is the loop condition in Delete() j <= currentSize and not j < currentSize?
Answer
The outer loop uses j <= currentSize to continue as long as the left child exists. Inside the loop, the right child comparison uses j < currentSize (strict less than) before accessing data[j+1]. If j == currentSize, there is only a left child — accessing data[j+1] would be out of bounds. So the outer loop allows j == currentSize (leaf with only left child) but the inner right-child check is strictly j < currentSize to prevent array overflow.
⚙️ Algorithm & Application (Q11–Q20)
Q11APPLYWhat is the time complexity of Insert, Delete, and HeapSort? Justify each with Big-O reasoning.
Answer
Insert: O(log n) — The new element bubbles up at most h levels, where h = height of a complete binary tree = ⌊log₂ n⌋. So at most log n comparisons/moves.

Delete: O(log n) — Same reasoning. The displaced last node bubbles down at most h levels = log n comparisons.

HeapSort: O(n log n) — Phase 1: n inserts × O(log n) each = O(n log n). Phase 2: n deletes × O(log n) each = O(n log n). Total = 2 × O(n log n) = O(n log n).

Q13GOTCHAWhat happens if you access data[j+1] inside the Delete loop without first checking j < currentSize?
Answer
Index out of bounds / undefined behaviour. When j == currentSize, the node at index j is the last valid element — there is no right child at j+1. Accessing data[j+1] would read beyond the used portion of the array. In the course's implementation (fixed-size array initialized with -1), this would compare against -1, which could accidentally trigger a swap. Always guard: if (j < currentSize && data[j] < data[j+1]).
Q14COMPARECompare a Max Heap and a Min Heap — definition, root value, and C# built-in behavior.
Answer
Max Heap: Parent ≥ all children. Root = maximum element. Delete always returns the largest value. The course's custom implementation is a max heap.

Min Heap: Parent ≤ all children. Root = minimum element. Delete returns the smallest value. C#'s PriorityQueue<TElement, TPriority> is a min heap — smallest priority number exits first.

To get a max-priority behavior from C#'s built-in, pass a custom comparer: Comparer<int>.Create((a,b) => b.CompareTo(a)).

Q16WHYHeapSort writes deleted values into the array from index n-1 downward. Why, not from index 0 upward?
Answer
Deletions from a max heap return values in descending order — largest first. To produce an ascending sorted array, the largest value must go to the end (index n-1), second-largest to n-2, and so on. Writing from right to left matches the descending output of deletions to produce ascending order. If you wrote from index 0, you'd get descending-sorted output — which is wrong for standard sort semantics.
Q17GOTCHAThe Heap constructor initializes all array values to -1. Why? What happens if you don't?
Answer
Initializing to -1 ensures that unused slots have a sentinel value that can't be confused with real heap data (assuming non-negative integers). During deletion, data[currentSize] = -1 clears the removed last slot — without this, the slot still holds the old value. If the loop in Delete() reads that stale value (especially if a bug causes j to overshoot), it could trigger a false comparison with a non-existent node. The -1 sentinel makes debugging easier and prevents false swaps.

Q19COMPARECompare HeapSort vs MergeSort — which is better and when?
Answer
HeapSort: O(n log n) all cases. Heap array needs O(n) extra space in the course's implementation (in-place variant exists with O(1) space). Not stable — equal elements may be reordered. Poor cache performance due to non-sequential memory access in the heap.

MergeSort: O(n log n) all cases. O(n) extra space. Stable — equal elements preserve order. Excellent cache performance, sequential memory access.

Choose MergeSort when stability matters or data is sequential (linked lists). Choose HeapSort when in-place sorting is needed with guaranteed O(n log n) worst case (unlike QuickSort's O(n²) worst).
Q20GOTCHACan a heap have duplicate values? Does that violate any property?
Answer
Yes, duplicates are fully valid. Neither the structural property nor the relational property prohibits equal values. The relational property says parent ≥ children (not strictly greater). So if a parent is 15 and a child is also 15, that satisfies the max heap condition. Duplicates are common in practice — for example, multiple tasks with the same priority level. The heap handles them correctly; the deletion order of equal elements is not guaranteed.
🟢 Real Interview Questions (Q21–Q40)
Q21INTERVIEWWhat data structure would you use to find the K largest elements in a stream of numbers efficiently?
Answer
Use a min-heap of size K. Process each incoming number: if heap.Count < K, enqueue it. Otherwise, if the number > heap.Peek() (current minimum), dequeue the minimum and enqueue the new number. After processing all elements, the heap contains the K largest.

var minHeap = new PriorityQueue<int,int>();
foreach (int n in stream) {
  if (minHeap.Count < K) minHeap.Enqueue(n, n);
  else if (minHeap.TryPeek(out _, out int min) && n > min) {
    minHeap.Dequeue(); minHeap.Enqueue(n, n);
  }
}
Time: O(n log K). Space: O(K).
Q22INTERVIEWHow would you implement an LRU Cache using a heap? What's the limitation?
Answer
A heap is a poor choice for LRU Cache. Here's why: LRU requires updating the priority of an existing element when it's accessed (re-queue it to "most recent"). Heap doesn't support O(1) arbitrary priority updates — you'd need to scan O(n) to find and remove the element, then re-insert it O(log n).

Better approach: Use LinkedList<T> + Dictionary<K, LinkedListNode>. Dictionary provides O(1) lookup, LinkedList provides O(1) move-to-front. Total: O(1) per get/put. A heap would make LRU O(n) per access — unusable at scale.
Q23INTERVIEWHow would you merge K sorted arrays into one sorted array efficiently?
Answer
Use a min-heap of size K (one entry per array). Each entry stores (value, array_index, element_index). Algorithm: initialize heap with the first element of each array. Then repeatedly: dequeue the minimum → add to result → enqueue the next element from that same array.

Time: O(n log K) where n = total elements. Space: O(K) for the heap. This is the optimal approach — used in database merge joins and external sorting.
Q24INTERVIEWExplain how Dijkstra's shortest path algorithm uses a priority queue and what happens if you use a regular queue instead.
Answer
Dijkstra's maintains a min-heap of unvisited nodes, keyed by their current shortest-known distance from the source. At each step, it dequeues the node with minimum distance (the greedy choice), relaxes its neighbors, and enqueues updated distances. This gives O((V + E) log V) with a binary heap.

With a regular queue: nodes are processed in FIFO order, not distance order. The greedy property breaks — you might process a far node before a nearby one. This produces incorrect shortest paths, not just slower ones.
Q25INTERVIEWWhat is the "heapify" operation and how does it build a heap in O(n) time instead of O(n log n)?
Answer
Heapify (Floyd's algorithm): Given an array, call down-heapify on every non-leaf node starting from the last non-leaf (index ⌊n/2⌋) down to the root. This builds a valid heap in O(n) time.

Why O(n) not O(n log n)? Most nodes are near the leaves (where down-bubbling is short or zero). Mathematically: the sum of work at all levels = O(n) when accounting for how many nodes exist at each height. In contrast, n individual insertions each bubble from leaf to root = O(n log n). Heapify is preferred when you have all data upfront.
Q26INTERVIEWFind the median of a stream of numbers in O(log n) per insertion and O(1) per query. What data structures do you use?
Answer
Use two heaps: a max-heap (left half) and a min-heap (right half). Invariant: max-heap.Count == min-heap.Count or max-heap.Count == min-heap.Count + 1.

Insert: route to correct heap based on value, then rebalance if sizes differ by more than 1 (move top of larger to smaller).

Query: if equal sizes → median = (maxHeap.Peek() + minHeap.Peek()) / 2. If left is larger → median = maxHeap.Peek().

This is the "two-heap median" pattern — classic interview question at Google/Amazon/Microsoft.
Q27INTERVIEWWhat is the time complexity of deleting an arbitrary element (not the root) from a heap? How would you do it?
Answer
O(n) total: O(n) to find the element (linear scan — no random access in heap), then O(log n) to restore heap property after removal.

Process: Find the element's index (O(n)). Replace it with the last element (like normal deletion). Decrement size. Then apply either up-heap or down-heap bubbling depending on whether the replacement is larger or smaller than the removed element (O(log n)).

To achieve O(log n) deletion of arbitrary elements, combine a heap with a HashMap: key → index. Update the map during every swap. This is how Fibonacci heaps and decrease-key operations work.
Q28INTERVIEWHow does a priority queue help in Huffman Encoding for data compression?
Answer
Huffman Encoding assigns shorter bit sequences to more frequent characters. Algorithm:
  1. Count frequency of each character → create leaf nodes.
  2. Insert all leaf nodes into a min-heap (priority = frequency).
  3. Repeatedly: dequeue the two lowest-frequency nodes, create a new parent node with their combined frequency, enqueue the parent.
  4. Repeat until one node remains — the root of the Huffman tree.
The priority queue ensures the two least frequent characters are always merged first, producing optimal prefix-free codes. Without a heap, finding the minimum each time would be O(n) per step — O(n²) total. With heap: O(n log n).
Q29INTERVIEWIs HeapSort stable? Why does this matter, and when would you avoid it?
Answer
HeapSort is NOT stable. Two elements with equal keys may swap their relative order during the heapify/down-heap process.

Why it matters: If you sort employees by department but want ties broken by employee ID (original order), an unstable sort may scramble the ID order within each department.

When to avoid: Any situation where maintaining the original order of equal elements is required — e.g., sorting database rows, UI list rendering, multi-key sorts where the secondary key is the original order. Use MergeSort or TimSort (C#'s Array.Sort and LINQ OrderBy are stable since .NET 4.5+) instead.
Q30INTERVIEWGiven a nearly sorted array (each element is at most K positions away from its sorted position), sort it in O(n log K). How?
Answer
Use a min-heap of size K+1:
  1. Insert first K+1 elements into the heap.
  2. For each remaining element i from K+1 to n-1: dequeue minimum → write to arr[i-K-1], enqueue arr[i].
  3. After loop: drain the heap into remaining positions.
Why it works: since each element is at most K positions away, the correct element for position i must be within the next K+1 elements. The heap always contains the K+1 candidates for the current output position.

Time: O(n log K) — optimal for this problem. Regular sort would be O(n log n).
Q31INTERVIEWWhat's the difference between Dequeue() and TryDequeue() in C#'s PriorityQueue? When should you use each?
Answer
Dequeue(): Removes and returns the element with lowest priority. Throws InvalidOperationException if the queue is empty. Use when you're certain the queue is non-empty, or you want the exception to bubble up.

TryDequeue(out element, out priority): Returns true and sets output params if queue has elements. Returns false silently if empty — no exception. Use in loops, in code where you're not sure if the queue has remaining elements, or in defensive programming: while (pq.TryDequeue(out var item, out var pri)) { ... }. This is safer and the idiomatic C# pattern for consuming a queue fully.
Q32INTERVIEWHow would you implement a task scheduler that can handle task deadlines and priorities simultaneously?
Answer
Use a composite priority: combine deadline and priority into a single comparable value. One approach: priority key = deadline × weight + priority_level. Or use two-level sorting with a custom comparer — first by deadline (earlier deadline = higher priority), ties broken by priority level.

var scheduler = new PriorityQueue<Task, (DateTime deadline, int priority)>(
  Comparer<(DateTime, int)>.Create((a, b) => {
    int d = a.Item1.CompareTo(b.Item1);
    return d != 0 ? d : a.Item2.CompareTo(b.Item2);
  }));
This is how real-time OS schedulers work — EDF (Earliest Deadline First) policy.
Q33INTERVIEWWhat is a d-ary heap and when would you prefer it over a binary heap?
Answer
A d-ary heap has d children per node instead of 2. Height = log_d(n), so down-heap operations do fewer swaps (shorter height) but compare more children per node (d comparisons per level).

Trade-off: Insert (up-heap) is faster with larger d — fewer levels to climb. Delete (down-heap) is slightly slower — more comparisons per level.

When preferred: When insertions are far more frequent than deletions (e.g., Dijkstra's with dense graphs). A 4-ary or 8-ary heap reduces cache misses on modern CPUs because more children fit in a single cache line — better memory locality than binary heap for large n.
Q34INTERVIEWGiven two sorted arrays of sizes m and n, find the (m+n)/2 th smallest element. Can you use a heap?
Answer
Heap approach (not optimal but correct): Merge both into a min-heap, then dequeue (m+n)/2 times. Time: O((m+n) log(m+n)). Space: O(m+n).

Better approach: Since both arrays are already sorted, use a two-pointer merge technique: advance the pointer of the smaller current element, count steps until you reach position (m+n)/2. Time: O(m+n). Space: O(1).

Optimal approach: Binary search on one array. Time: O(log(min(m,n))). Space: O(1).

The heap approach is valid but not the interview-expected optimal solution here.
Q35INTERVIEWWhat is a Fibonacci Heap and why is it theoretically superior for Dijkstra's algorithm?
Answer
A Fibonacci Heap is a lazy heap with amortized O(1) insert, O(1) decrease-key, and O(log n) delete-min. This makes Dijkstra's run in O(V log V + E) — optimal for sparse graphs.

Binary heap Dijkstra's: O((V + E) log V). With Fibonacci heap: O(E + V log V). For dense graphs where E >> V, this is a significant improvement.

Why not used in practice: Fibonacci heaps have enormous constant factors and complex implementation. Cache performance is poor. For real-world graph sizes, a well-implemented binary heap outperforms Fibonacci heaps despite worse theoretical complexity.
Q36INTERVIEWHow would you check if a given binary tree is a valid Max Heap?
Answer
Two checks required:

1. Structural check (complete binary tree): BFS level-order traversal. Count total nodes n. In a complete binary tree, if you encounter a null child, all subsequent nodes in BFS order must also be null. Flag any non-null node after first null as violation.

2. Relational check: For every non-leaf node, verify node.value ≥ leftChild.value AND node.value ≥ rightChild.value. Do this recursively for all nodes.

Both checks must pass. Time: O(n). Space: O(h) for recursion stack or O(n) for BFS queue.
Q37INTERVIEWDesign a system to always return the most recently added item that hasn't been processed, and items with equal priority should be processed in FIFO order. What data structure?
Answer
Use a priority queue with tie-breaking by insertion order. Key = (priority, sequence_number) where sequence_number is a monotonically increasing counter assigned at insertion.

int seq = 0;
var pq = new PriorityQueue<Task, (int priority, int seq)>(
  Comparer<(int,int)>.Create((a,b) => {
    int p = a.Item1.CompareTo(b.Item1);
    return p != 0 ? p : a.Item2.CompareTo(b.Item2);
  }));
pq.Enqueue(task, (priority, seq++));
Higher priority exits first. Equal priority → smaller seq exits first (FIFO). This is the standard stable priority queue pattern.
Q38INTERVIEWWhat are the main real-world applications of heaps in systems programming and databases?
Answer
  • OS Process Scheduling: CPU schedulers (CFS in Linux) use red-black trees / heaps to maintain runnable processes sorted by virtual runtime.
  • Network QoS: Packet queues prioritize traffic (VOIP over FTP) using weighted priority queues.
  • Database Query Planning: Top-N queries use a heap to find the N largest rows without full sort.
  • Event-driven simulation: Next-event simulation engines use a min-heap of (timestamp, event) — process earliest event first.
  • Memory allocators: Some free-list implementations use a heap to always allocate from the best-fit or smallest-fit block.
  • Timer wheels: Kernel timers use heaps to efficiently fire the next-expiring timer.
Q39INTERVIEWWhat is the space complexity of the HeapSort implementation in the course? How could you reduce it to O(1) extra space?
Answer
The course implementation creates a new Heap object with a separate array — O(n) extra space.

In-place HeapSort (O(1) space):
Phase 1: Call heapify (Floyd's algorithm) directly on the input array — convert it into a max-heap in-place. O(n).
Phase 2: Repeatedly swap arr[0] (root/max) with arr[lastUnsortedIndex], reduce heap size by 1, then down-heapify at index 0. This sorts the array in-place.

In-place HeapSort is O(n log n) time and O(1) extra space — the same array serves as both heap storage and sorted output. This is the production-quality version.
Q40INTERVIEWA colleague says "I'll just use Array.Sort() — it's already O(n log n), why learn HeapSort?" How do you respond?
Answer
Valid point for most cases — Array.Sort() (TimSort) is usually faster in practice. But HeapSort has specific advantages:

  • Worst-case guarantee: HeapSort is O(n log n) worst case. QuickSort (used in Array.Sort for primitives) is O(n²) worst case on adversarial inputs.
  • Understanding heap: Heap Sort teaches the heap data structure itself — which is essential for priority queues, Dijkstra's, scheduling. The sort is a by-product.
  • Streaming/online: Heap-based selection (top-K) doesn't require all data upfront. TimSort does.
  • Embedded/real-time systems: Where predictable worst-case is required and random pivots aren't acceptable.
Agree to use Array.Sort() in production, but understand when heaps are the right tool.
