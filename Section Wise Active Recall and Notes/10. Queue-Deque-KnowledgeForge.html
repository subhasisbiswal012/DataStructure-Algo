Queue & Deque

Queue Data Structure

1. What is a Queue?

💡 Think of a queue at a movie ticket counter. The person who arrived first gets served first. Nobody cuts the line from behind.

A Queue is a linear data structure that follows the FIFO principle — First In, First Out. The first element you add is the first element you take out. You insert at one end called the Rear, and you remove from the other end called the Front.

  • Enqueue — insert an element at the rear
  • Dequeue — remove and return the element at the front
  • Peek / Front — see the front element without removing it
  • IsEmpty — check if no elements exist
  • Count — how many elements are currently in the queue
📌 FIFO = First In, First Out. Whatever enters the queue first, leaves first. The newest arrival must wait at the back — no shortcuts.

3. C# Built-in Queue<T>

💡 C# provides Queue<T> in System.Collections.Generic. It's generic, type-safe, and backed by a circular array internally.
csharp
using System.Collections.Generic;

Queue<string> queue = new Queue<string>();

queue.Enqueue("Alice");   // rear: Alice
queue.Enqueue("Bob");     // rear: Alice → Bob
queue.Enqueue("Charlie"); // rear: Alice → Bob → Charlie

string front = queue.Peek();     // "Alice" — does NOT remove
string served = queue.Dequeue(); // "Alice" — REMOVES from front

bool has = queue.Contains("Bob"); // true — searches entire queue
int count = queue.Count;          // 2
bool empty = queue.Count == 0;    // false

queue.Clear();           // empties everything
string[] arr = queue.ToArray(); // copy to array (front-to-back order)

4. Why a Circular Array? — Internals of Queue<T>

💡 Without a circular array, dequeuing from the front of a regular array would shift all elements left — costing O(n) per dequeue. A circular array avoids all shifting.

The Problem with a Plain Array

Imagine you dequeue from index 0 of a regular array. You then need to slide every remaining element one position to the left to keep the front at index 0. That's O(n) per dequeue — awful for a queue that dequeues constantly.

What is a Circular Array?

A circular array treats the array as if it wraps around like a ring. Instead of moving elements when you dequeue, you just move a front pointer forward. When rear reaches the end of the array, it wraps back to index 0 using the modulo operator: rear = (rear + 1) % capacity.

Circular Array Queue — C# Code

csharp — circular array queue implementation
public class CircularArrayQueue
{
    private int[] _data;
    private int _front = 0;   // index of the front element
    private int _rear  = 0;   // index where next element will go
    private int _count = 0;
    private int _capacity;

    public CircularArrayQueue(int capacity)
    {
        _capacity = capacity;
        _data     = new int[capacity];
    }

    public bool IsEmpty => _count == 0;
    public bool IsFull  => _count == _capacity;
    public int  Count   => _count;

    public void Enqueue(int item)
    {
        if (IsFull) throw new InvalidOperationException("Queue is full.");
        _data[_rear] = item;
        _rear  = (_rear + 1) % _capacity;  // ← key: wrap around!
        _count++;
    }

    public int Dequeue()
    {
        if (IsEmpty) throw new InvalidOperationException("Queue is empty.");
        int val = _data[_front];
        _front = (_front + 1) % _capacity; // ← key: move front forward
        _count--;
        return val;
    }

    public int Peek() => IsEmpty
        ? throw new InvalidOperationException("Queue is empty.")
        : _data[_front];
}
📌 The magic line: rear = (rear + 1) % capacity. When rear = capacity-1 and you add 1, modulo wraps it back to 0 — as if the array is a circle.

5. Array-Based vs Linked List-Based Queue

💡 Both implement the same FIFO behaviour. The difference is under the hood — and that matters for memory and performance.
Array-Based (C# Queue<T>)
  • Backed by circular array — O(1) enqueue & dequeue
  • Cache-friendly — contiguous memory
  • Occasional O(n) resize when capacity exceeded
  • Some wasted capacity (empty slots)
  • Less memory per element — no pointer overhead
Linked List-Based
  • O(1) always — no resize, no spikes
  • No wasted capacity — one node per element
  • Pointer overhead per node (+8 bytes on 64-bit)
  • Scattered memory — cache misses more likely
  • Predictable latency — preferred for real-time systems

Linked List Queue — C# Implementation

csharp
// C# LinkedList<T> used as a Queue (built-in doubly linked list)
LinkedList<int> queue = new LinkedList<int>();

// Enqueue → add at tail (AddLast)
queue.AddLast(10);
queue.AddLast(20);
queue.AddLast(30);

// Peek → read front without removing
int front = queue.First.Value;  // 10

// Dequeue → read front then remove (RemoveFirst)
int served = queue.First.Value; // 10
queue.RemoveFirst();             // O(1) — no traversal needed

// All enqueue (AddLast) and dequeue (RemoveFirst) are O(1)
📌 For a linked list queue: AddLast = Enqueue, RemoveFirst = Dequeue. Both are O(1) because we have direct access to both ends via head (front) and tail (rear).

Deque (Double-Ended Queue)

1. What is a Deque?

💡 "Deque" is short for Double-Ended Queue. If a Queue is a one-way street, a Deque is a two-way street. You can add or remove from either end.

A Deque (pronounced "deck") is a data structure that generalises both the Queue and the Stack. It allows insertion and deletion at both the front and the rear — giving you full control over both ends.

  • AddFront — insert at the front
  • AddRear — insert at the rear
  • RemoveFront — remove from the front
  • RemoveRear — remove from the rear
  • PeekFront / PeekRear — read either end without removing
📌 A Deque is a superset: it can behave as a Queue (FIFO) by only using AddRear + RemoveFront, or as a Stack (LIFO) by only using AddFront + RemoveFront.

2. Deque in Modern C# — LinkedList<T>

💡 C# doesn't have a class literally named "Deque". But LinkedList<T> in System.Collections.Generic is the official, built-in way to get full Deque behaviour. It's a doubly linked list — so O(1) at both ends naturally.
csharp
using System.Collections.Generic;

LinkedList<int> deque = new LinkedList<int>();

// ── ADD ──────────────────────────────────
deque.AddLast(10);   // deque: [10]         ← add to rear
deque.AddLast(20);   // deque: [10, 20]
deque.AddFirst(5);   // deque: [5, 10, 20]  ← add to front
deque.AddFirst(1);   // deque: [1, 5, 10, 20]

// ── PEEK ─────────────────────────────────
int peekFront = deque.First.Value;  // 1  (does NOT remove)
int peekRear  = deque.Last.Value;   // 20 (does NOT remove)

// ── REMOVE ───────────────────────────────
deque.RemoveFirst();  // removes 1, deque: [5, 10, 20]
deque.RemoveLast();   // removes 20, deque: [5, 10]

// ── INFO ─────────────────────────────────
int count = deque.Count;          // 2
bool empty = deque.Count == 0;    // false
bool has   = deque.Contains(10);  // true

// ── UTILITY ──────────────────────────────
deque.Remove(10);    // remove specific value (O(n) search)
deque.Clear();       // remove all

3. Why Deque? Real-World Uses

💡 A Deque is used whenever you need fast operations at both ends — not just one. The moment you need to "add to front" efficiently, a plain Queue fails you. Deque steps in.
↩️↪️
Undo + Redo
Undo pops from front. Redo pushes back to front. Deque handles both naturally.
🌐
Browser History
New pages added to front. Back/Forward navigates both ends. Deque is perfect.
📋
Sliding Window Max
Classic algorithm (e.g., LeetCode 239) — front holds max, expired elements dropped from rear and front.
📅
Task Scheduling
High-priority tasks inserted at front; normal tasks at rear. Priority without a full priority queue.
🔁
Palindrome Check
Compare front and rear, pop both sides inward. O(n) check using deque.
🎮
Game AI Pathfinding
0-1 BFS uses a deque — weight-0 edges added to front, weight-1 to rear.
🖥️
Work-Stealing Queues
JVM thread pools steal tasks from the rear of another thread's deque. Both ends are live.
📦
Input Buffer / Text Editors
Cursor movements insert/delete at arbitrary positions — deque enables efficient front/back editing.

4. Before LinkedList<T> — How Deque Was Built

💡 Before C# had LinkedList<T> as a clean Deque, developers had to build one manually. The two approaches were: a circular doubly linked list from scratch, or a pair of stacks.

Approach 1 — Two Stacks as a Deque

Use two stacks — frontStack and rearStack. Each stack covers one end. When one runs out, rebalance by reversing the other into it.

csharp — deque from two stacks (old-school approach)
public class DequeFromTwoStacks<T>
{
    private Stack<T> _front = new Stack<T>();
    private Stack<T> _rear  = new Stack<T>();

    public void AddFront(T item) => _front.Push(item);
    public void AddRear(T item)  => _rear.Push(item);

    public T RemoveFront()
    {
        if (_front.Count == 0)
            Rebalance(_rear, _front);      // move rear elements to front
        if (_front.Count == 0) throw new InvalidOperationException("Empty");
        return _front.Pop();
    }

    public T RemoveRear()
    {
        if (_rear.Count == 0)
            Rebalance(_front, _rear);      // move front elements to rear
        if (_rear.Count == 0) throw new InvalidOperationException("Empty");
        return _rear.Pop();
    }

    private void Rebalance(Stack<T> source, Stack<T> dest)
    {
        int half = source.Count / 2;
        // move half of source into a temp, other half into dest, then back
        var temp = new Stack<T>();
        while (source.Count > half) temp.Push(source.Pop());
        while (source.Count > 0)    dest.Push(source.Pop());
        while (temp.Count > 0)      source.Push(temp.Pop());
    }
}
⚠️ Two-stack Deque gives O(1) amortised for all operations but individual rebalance calls cost O(n). Not suitable for real-time systems.

Approach 2 — Custom Doubly Linked List (Manual Node Class)

Before LinkedList<T> was available (or trusted), developers wrote a raw doubly linked list themselves — a Node class with prev and next, plus a wrapper class managing head and tail.

csharp — manual doubly linked deque (pre-LinkedList era)
public class DequeNode<T>
{
    public T Value;
    public DequeNode<T> Prev, Next;
    public DequeNode(T v) { Value = v; }
}

public class ManualDeque<T>
{
    private DequeNode<T> _head, _tail;
    public int Count { get; private set; }

    public void AddFront(T item)
    {
        var node = new DequeNode<T>(item);
        if (_head == null) { _head = _tail = node; }
        else { node.Next = _head; _head.Prev = node; _head = node; }
        Count++;
    }

    public void AddRear(T item)
    {
        var node = new DequeNode<T>(item);
        if (_tail == null) { _head = _tail = node; }
        else { _tail.Next = node; node.Prev = _tail; _tail = node; }
        Count++;
    }

    public T RemoveFront()
    {
        if (_head == null) throw new InvalidOperationException();
        T val = _head.Value;
        _head = _head.Next;
        if (_head != null) _head.Prev = null; else _tail = null;
        Count--;
        return val;
    }

    public T RemoveRear()
    {
        if (_tail == null) throw new InvalidOperationException();
        T val = _tail.Value;
        _tail = _tail.Prev;
        if (_tail != null) _tail.Next = null; else _head = null;
        Count--;
        return val;
    }
}
📌 Today, just use LinkedList<T> — it is this manual deque, already implemented, tested, and maintained by Microsoft. No reason to reinvent it.

Q01DEFWhat is a Queue? Define it with FIFO and state what operations are allowed.
Answer
A Queue is a linear data structure following FIFO — First In, First Out. The element that was added earliest is always the first to be removed. You can only insert at one end (the rear) and remove from the other end (the front). Core operations: Enqueue (add to rear), Dequeue (remove from front), Peek (read front without removing), Count, IsEmpty.
Analogy: A line at a cinema ticket counter — first person in line is first served.
Q02WHYWhy does C#'s Queue<T> internally use a circular array instead of a plain array?
Answer
A plain array requires shifting all elements left every time you dequeue from the front — that's O(n) per dequeue. A circular array instead keeps a front pointer that advances on dequeue, and a rear pointer that wraps to index 0 using (rear + 1) % capacity when it hits the end. No element is ever moved. Both enqueue and dequeue become O(1). Memory is also reused — dequeued slots at the front are reclaimed for new elements.
Key line: rear = (rear + 1) % capacity — this is the wrap-around magic.
Q03APPLYWrite C# code to process print jobs from a Queue. Enqueue "Doc1", "Doc2", "Doc3", then dequeue and print each.
Answer
csharp
Queue<string> printQueue = new Queue<string>();
printQueue.Enqueue("Doc1");
printQueue.Enqueue("Doc2");
printQueue.Enqueue("Doc3");

while (printQueue.Count > 0)
    Console.WriteLine("Printing: " + printQueue.Dequeue());
// Output: Doc1, Doc2, Doc3 (in that order — FIFO)
Q04GOTCHAWhat exception does Queue<T>.Dequeue() or Peek() throw on an empty queue? How do you avoid it?
Answer
Both throw InvalidOperationException when the queue is empty. Avoid it by using: TryDequeue(out T result) — returns false instead of throwing; or TryPeek(out T result); or explicitly checking queue.Count > 0 before calling Dequeue/Peek.
Best practice: Always use TryDequeue in production code where empty queue is a valid scenario.
Q05COMPARECompare Queue and Stack — FIFO vs LIFO. Give a real-world example for each.
Answer
Queue (FIFO): First added = first removed. Example: printer queue — first document sent is first printed. First-come-first-served.

Stack (LIFO): Last added = first removed. Example: browser Back button — the most recently visited page is where "Back" takes you. Most recent first.
Q06APPLYImplement Breadth-First Search (BFS) using a Queue in C#. Sketch the approach.
Answer
csharp — BFS skeleton
void BFS(int start, Dictionary<int, List<int>> graph)
{
    var visited = new HashSet<int>();
    var queue   = new Queue<int>();

    queue.Enqueue(start);
    visited.Add(start);

    while (queue.Count > 0)
    {
        int node = queue.Dequeue();
        Console.Write(node + " ");

        foreach (int neighbour in graph[node])
        {
            if (!visited.Contains(neighbour))
            {
                visited.Add(neighbour);
                queue.Enqueue(neighbour);  // explore level by level
            }
        }
    }
}
Why Queue? BFS processes nodes level-by-level. The queue ensures nodes are visited in the order they were discovered (FIFO = level order).
Q07SEQUENCEEnqueue 5, 10, 15 into a queue. Then Dequeue once, Peek once, Dequeue again. What are the results?
Answer
Queue after Enqueue(5, 10, 15): [5, 10, 15] — 5 is at front.
Dequeue() → returns 5. Queue: [10, 15]
Peek() → returns 10. Queue unchanged: [10, 15]
Dequeue() → returns 10. Queue: [15]
Q08WHYWhy is Enqueue O(1) amortised rather than strictly O(1) in Queue<T>?
Answer
Queue<T> internally uses a circular array. When the array fills up, it allocates a new array of double the size and copies all elements — a single O(n) operation. However, because the array doubles each time, resizes become exponentially rare. Averaged across all enqueue operations, the cost per enqueue is O(1) — this is amortised O(1). Any single enqueue could be O(n) if it triggers a resize.
Q09APPLYUsing only a Queue, reverse the integers in [1, 2, 3, 4, 5] to get [5, 4, 3, 2, 1]. How?
Answer
You cannot reverse a queue with only a queue. You need a stack (or recursion) to reverse.
csharp
Queue<int> q = new Queue<int>(new[] {1,2,3,4,5});
Stack<int> s = new Stack<int>();
while (q.Count > 0) s.Push(q.Dequeue()); // dump into stack
while (s.Count > 0) q.Enqueue(s.Pop());  // pull back (reversed)
// q is now [5,4,3,2,1]
Gotcha: A single queue cannot self-reverse — that's a key difference from a stack.
Q10GOTCHADoes foreach on a Queue<T> dequeue elements? What order does it iterate?
Answer
No — foreach does NOT dequeue. The queue remains intact after iteration. It iterates from front to rear (oldest to newest). If you enqueue 1, 2, 3 and do foreach, you see 1, 2, 3 — the same order as dequeuing would give, but nothing is removed.
Gotcha: Modifying the queue (Enqueue/Dequeue) during foreach throws InvalidOperationException.
Q11APPLYImplement a Queue using two Stacks. What is the amortised time complexity of each operation?
Answer
csharp
public class QueueFromStacks<T>
{
    Stack<T> _inbox = new(), _outbox = new();
    public void Enqueue(T item) => _inbox.Push(item);
    public T Dequeue()
    {
        if (_outbox.Count == 0)
            while (_inbox.Count > 0)
                _outbox.Push(_inbox.Pop());  // transfer reverses → FIFO
        if (_outbox.Count == 0) throw new InvalidOperationException();
        return _outbox.Pop();
    }
}
Complexity: Enqueue = O(1). Dequeue = O(1) amortised (each element transferred at most once).
Q12COMPARECompare array-based queue vs linked list-based queue. When would you choose each?
Answer
Array-based (Queue<T>): Cache-friendly, less memory overhead, occasional O(n) resize. Use for general purpose — most applications.

Linked list-based (LinkedList<T>): Strictly O(1) enqueue/dequeue always, no resize, but each node has pointer overhead and is scattered in memory. Use when you need guaranteed worst-case O(1) — real-time systems, trading engines, embedded software.
Q13WHYWhy is a Queue used in BFS instead of a Stack? What happens if you use a Stack instead?
Answer
BFS explores nodes level by level. A queue ensures the first discovered nodes are visited first (FIFO = level order). If you use a Stack instead, you process the most recently discovered node first — which gives you DFS (Depth-First Search), not BFS. The choice of data structure literally defines which traversal algorithm you're running.
Q14DEFWhat is a Priority Queue? How is it different from a regular Queue?
Answer
A regular Queue is FIFO — first in, first out regardless of importance. A Priority Queue dequeues the element with the highest priority first — regardless of insertion order. C# has PriorityQueue<TElement, TPriority> (added in .NET 6). Internally it uses a min-heap. Enqueue with a priority, Dequeue returns the element with the lowest priority number (highest priority).
Example: Hospital ER — critical patients (priority 1) are seen before minor injuries (priority 3) even if they arrived later.
Q15APPLYWhat is TrimExcess() on Queue<T> and when would you use it?
Answer
TrimExcess() reduces the internal array's capacity to match the current Count. After a resize, the array might have 2n capacity but only n elements. Calling TrimExcess releases the wasted slots. Use it when: you've finished filling a queue and won't add more elements, and memory pressure is a concern. Time: O(n) — must copy all elements to a smaller array.
Q16SEQUENCEIn a circular array queue with capacity 5, front=3, rear=1, what are the valid indices currently occupied?
Answer
Indices occupied: 3, 4, 0 (wraps around). The queue "wraps" from index 4 back to 0 before reaching rear=1. So the elements are at indices 3, 4, 0 — that's 3 elements. Rear=1 means the next Enqueue would go to index 1. This is exactly why a circular array can reuse front slots that were dequeued.
Q17GOTCHAIn a full circular array queue (front == rear), how do you distinguish "full" from "empty" since both states could have front == rear?
Answer
Classic problem. Three solutions: (1) Maintain a count variable — empty if count==0, full if count==capacity. This is what C#'s Queue<T> does. (2) Keep one slot always empty — full when (rear+1)%capacity==front. (3) Use a boolean flag — track "was last operation enqueue or dequeue?" C# uses approach #1 (count variable) because it also provides O(1) Count property.
Q18APPLYGiven a stream of integers, find the sliding window maximum of size k using a Deque. Sketch the algorithm.
Answer
csharp — sliding window max (classic Deque use)
int[] SlidingWindowMax(int[] nums, int k)
{
    var deque  = new LinkedList<int>(); // stores indices
    var result = new int[nums.Length - k + 1];
    int ri = 0;

    for (int i = 0; i < nums.Length; i++)
    {
        // remove elements outside window from front
        while (deque.Count > 0 && deque.First.Value < i - k + 1)
            deque.RemoveFirst();

        // remove smaller elements from rear (they can never be max)
        while (deque.Count > 0 && nums[deque.Last.Value] < nums[i])
            deque.RemoveLast();

        deque.AddLast(i);

        if (i >= k - 1)
            result[ri++] = nums[deque.First.Value]; // front = max of window
    }
    return result;
}
// Input: [1,3,-1,-3,5,3,6,7], k=3 → Output: [3,3,5,5,6,7]
Why Deque? You need to add/remove from both ends: drop old indices from front, drop smaller candidates from rear. No other single structure supports both ends in O(1).

Q19DEFWhat is a Deque? How is it different from both a Queue and a Stack?
Answer
A Deque (Double-Ended Queue) allows insert and remove at both the front and the rear. A Queue only allows enqueue at rear and dequeue at front (FIFO). A Stack only allows push/pop at the top (LIFO). A Deque is a superset — it can simulate both. It doesn't enforce FIFO or LIFO — the caller decides which end to use.
Q20WHYWhy does C# not have a dedicated Deque class, and what do you use instead?
Answer
C# does not have a class named "Deque" in the standard library, but LinkedList<T> in System.Collections.Generic acts as a perfect Deque — it has AddFirst, AddLast, RemoveFirst, RemoveLast all in O(1). It's a doubly linked list so both ends are instantly accessible. Some developers also use third-party libraries (e.g., C5 collections) that have an explicit Deque type.
Q21APPLYCheck if a string is a palindrome using a Deque (LinkedList in C#). Write the code.
Answer
csharp
bool IsPalindrome(string s)
{
    var deque = new LinkedList<char>(s);
    while (deque.Count > 1)
    {
        if (deque.First.Value != deque.Last.Value) return false;
        deque.RemoveFirst();
        deque.RemoveLast();
    }
    return true;
}
// "racecar" → true, "hello" → false
Why Deque? Comparing and removing from both ends simultaneously — only a Deque makes both operations O(1).
Q22GOTCHAWhat is the difference between Remove(T value) and Remove(LinkedListNode) in LinkedList<T>? Which is faster and why?
Answer
Remove(T value) — must scan the list to find the first node with this value. O(n) because it's a linear search.

Remove(LinkedListNode) — you already have the node reference. Just update its neighbours' pointers. O(1) because no search is needed — pointer rewiring only.

If you're frequently removing specific elements, store the node reference (from AddFirst/AddLast return value) and use the node overload.
Q23SEQUENCEOn a LinkedList<T> acting as a Deque, perform: AddLast(10), AddFirst(5), AddLast(20), RemoveFirst(), PeekRear(). What is the result?
Answer
Start: []
AddLast(10) → [10]
AddFirst(5) → [5, 10]
AddLast(20) → [5, 10, 20]
RemoveFirst() → removes 5. List: [10, 20]
PeekRear → deque.Last.Value → 20 (list unchanged: [10, 20])
Q24COMPAREHow was a Deque implemented before LinkedList<T> was available? What are the trade-offs?
Answer
Two Stacks: One stack for front, one for rear. Amortised O(1) per operation, but rebalancing causes occasional O(n) spikes. Simple to implement.

Manual doubly linked list: Custom Node class with prev + next, head and tail tracked. Strictly O(1) all operations — same performance as LinkedList<T> today. More code to write and maintain, and no battle-tested standard library backing.

Today: Just use LinkedList<T> — it's the same as the manual DLL, already tested.
Q25APPLYImplement a browser history (Back/Forward) using LinkedList<T> as a Deque. Describe the data model.
Answer
csharp — browser history model
public class BrowserHistory
{
    private LinkedList<string> _history = new();
    private LinkedListNode<string> _current;

    public void Visit(string url)
    {
        // remove all "forward" history (nodes after current)
        while (_current != null && _current.Next != null)
            _history.RemoveLast();

        _history.AddLast(url);
        _current = _history.Last;
    }

    public string Back()
    {
        if (_current?.Prev == null) return _current?.Value ?? "No history";
        _current = _current.Prev;
        return _current.Value;
    }

    public string Forward()
    {
        if (_current?.Next == null) return _current?.Value ?? "No history";
        _current = _current.Next;
        return _current.Value;
    }
}
Q26WHYWhy is LinkedList<T> better than a circular array for Deque operations?
Answer
A circular array supports O(1) add/remove at both ends only if it never resizes — but it has a fixed capacity. When full, it must resize (O(n)). LinkedList<T> has no capacity limit — each node is allocated on demand. AddFirst/AddLast/RemoveFirst/RemoveLast are strictly O(1) always — no surprise resizes. The trade-off: each node carries pointer overhead (prev + next) and is scattered in memory, causing more cache misses than a contiguous array.
Q27GOTCHAWhat does deque.First return if the LinkedList is empty? Does it throw?
Answer
LinkedList<T>.First returns null if the list is empty — it does NOT throw an exception. The danger is that accessing .Value on the null node will throw a NullReferenceException. Always check before accessing: if (deque.First != null) { var v = deque.First.Value; } — or use the null-conditional operator: deque.First?.Value.
Contrast with Queue: Queue<T>.Peek() throws InvalidOperationException on empty. LinkedList<T>.First is null — a different failure mode.
Q28APPLYHow do work-stealing thread pools (like .NET's ThreadPool) use a Deque internally?
Answer
Each worker thread has its own local Deque of tasks. The owning thread adds tasks to and takes tasks from its front (LIFO — for better cache locality). When a thread runs out of work, it "steals" tasks from another thread's rear end (FIFO — takes older tasks). This lets threads work without locking (mostly), and idle threads help busy threads. The Deque is essential because both ends need to be independently accessible and fast.
Q29COMPAREWhat is the space complexity of LinkedList<T> compared to Queue<T> for the same n elements? Which uses more memory?
Answer
Both are O(n) space overall, but LinkedList<T> uses more memory per element. Each LinkedListNode stores: the value + a prev pointer + a next pointer (typically 3 × 8 = 24 bytes on 64-bit just for the node object, plus GC overhead). Queue<T> stores just the values in a contiguous array — the only overhead is the array wrapper itself, shared across all elements. For large n with value types (int, struct), Queue<T> is significantly more memory-efficient.
Q30APPLYAn interviewer asks: "Implement a Stack using only a Queue (single queue, not two)." Walk through the approach.
Answer
csharp — stack using single queue
public class StackFromOneQueue<T>
{
    private Queue<T> _q = new();

    // On every Push: enqueue, then rotate the queue so new item is at front
    public void Push(T item)
    {
        _q.Enqueue(item);
        int size = _q.Count;
        while (size-- > 1)
            _q.Enqueue(_q.Dequeue());  // rotate: move front items to back
    }

    public T Pop()  => _q.Dequeue();   // front is always the last-pushed item
    public T Peek() => _q.Peek();
}
// Push is O(n) — rotates n-1 times
// Pop/Peek is O(1)
Key insight: After each Push, rotate all n-1 existing items to the back. This puts the newest item at the front — which is what Pop() needs for LIFO behaviour.
