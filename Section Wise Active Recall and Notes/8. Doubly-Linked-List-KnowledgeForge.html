Circular Linked List — C# Custom Implementation

🔄 What is a Circular Linked List?

💡 Think of it like players sitting at a poker table. After the last player, the turn goes back to the first — there's no "end", it loops forever.
  • A Circular Linked List is a chain of nodes where the last node's next pointer points back to the first node — not null.
  • Every node has two parts: element (the data) and next (address of next node).
  • The list has a head (first node) and a tail (last node), but tail.next ≠ null — it equals head.
  • A single node in a circular list points to itself (its own address in next).

⚠️ You cannot use while(p != null) to traverse a circular linked list — tail.next points back to head, so p will never be null. You'll loop forever. Always use a counter: while(i < size).

🧱 The Node Class

💡 A node is a small container — it holds one piece of data and a forwarding address to the next container in line.
csharp
class Node
{
    public int element;   // stores the data value
    public Node next;     // stores the reference (address) of the next node

    public Node(int e, Node n)
    {
        element = e;
        next = n;
    }
}

addLast(int e) Time: O(1) Space: O(1) Add a new node at the END of the list
💡 Think of it like joining the back of a circular queue — you get in line behind the last person, and the last person's "next turn" card now points to you. The tail pointer moves to you.
Time — Empty case O(1) Direct pointer assignments only — no loop
Time — Non-empty case O(1) We hold a direct tail reference — no traversal needed
Space O(1) One new node allocated regardless of list size

Case 1 — Empty List

When size = 0, the new node becomes both head and tail, and its next points to itself (circular).

Case 2 — Non-Empty List

The new node slots in between the current tail and head, then becomes the new tail.

csharp
public void addLast(int e)
{
    Node newest = new Node(e, null);
    if(IsEmpty())
    {
        newest.next = newest;   // points to itself — one-node circle
        head = newest;
    }
    else
    {
        newest.next = tail.next;  // new node → head (since tail.next = head)
        tail.next = newest;       // old tail → new node
    }
    tail = newest;   // tail always moves to the new node
    size++;
}
addFirst(int e) Time: O(1) Space: O(1) Add a new node at the BEGINNING of the list
💡 Imagine cutting in line at the front of a circular queue — tail's "next turn" card needs to be rewritten to point to you, and you point to whoever used to be first.
Time O(1) Both head and tail pointers are directly accessible — no traversal
Space O(1) One new node allocated, three pointer assignments
Key constraint Order matters tail.next = newest must run before head = newest — or you lose the head reference

Non-Empty Case (the key scenario)

⚠️ The order matters: tail.next = newest must happen BEFORE head = newest. Once you move head, you lose the reference needed to write tail.next correctly.

addAnywhere(int e, int pos) Time: O(n) Space: O(1) Insert at any position in the middle
💡 Like inserting a new player into a specific seat at the poker table — you stop one seat BEFORE the target, break the link, and wire the new player in between.
Time — Best case (pos ≈ 1) O(1) Loop runs 0 iterations if inserting near the head
Time — Worst case (pos ≈ n) O(n) Must traverse up to pos−1 nodes to reach insertion point
Space O(1) One new node + two pointer rewires regardless of n
csharp
public void addAnywhere(int e, int pos)
{
    if(pos <= 0 || pos >= size)
    {
        Console.WriteLine("Invalid Entry");
        return;
    }
    Node newest = new Node(e, null);
    Node p = head;
    int i = 1;
    while(i < pos - 1)   // traverse to node BEFORE insertion point
    {
        p = p.next;
        i++;
    }
    newest.next = p.next;  // ① new node → what p used to point to
    p.next = newest;       // ② p → new node
    size++;
}
removeFirst() Time: O(1) Space: O(1) Remove the HEAD node
💡 The first player leaves the table. The player who used to be second becomes the new first. But the last player still needs to know who to pass to — so tail.next must be updated to the new head.
Time O(1) Uses only head and tail — no iteration needed
Space O(1) No extra memory — just pointer reassignments
Edge case cost O(1) Even the empty-list cleanup (head=null, tail=null) is constant time
csharp
public int removeFirst()
{
    if(IsEmpty()) { Console.WriteLine("Empty"); return -1; }
    int e = head.element;          // save value to return
    tail.next = head.next;         // rewire tail to skip deleted head
    head = head.next;              // advance head
    size--;
    if(IsEmpty()) { head = null; tail = null; }  // cleanup if now empty
    return e;
}
removeLast() Time: O(n) Space: O(1) Remove the TAIL node
💡 The last player at the table leaves. But you can't jump straight to the last player — you have to walk around the entire table to find the second-to-last person and tell them "you're the new last now, point back to head."
⚠️ We can't navigate backwards in a singly-linked list, so we must traverse to size−1 (the node just before tail) and update from there.
Time O(n) Must walk from head to node at size−1 — no shortcut exists in a singly-linked list
Space O(1) Only a traversal pointer p and counter — no extra allocation
Why not O(1)? No prev ptr A doubly-linked list would give O(1) via tail.prev — singly-linked pays the traversal cost
csharp
public int removeLast()
{
    if (IsEmpty()) { Console.WriteLine("Empty"); return -1; }
    Node p = head;
    int i = 1;
    while(i < length() - 1)  // stop at second-to-last node
    {
        p = p.next;
        i++;
    }
    tail = p;           // p is the new tail
    p = p.next;         // p now = old tail
    tail.next = head;   // critical: new tail must point back to head (circular!)
    int e = p.element;  // save deleted value
    size--;
    return e;
}

💡 Removing a player mid-table — you stop at the player before the one leaving, and rewire that player's "pass to" card to skip over the one leaving.
Time — Best case O(1) If position ≈ 1, the loop body barely runs
Time — Worst case O(n) Must traverse up to pos−1 nodes; worst case is near the tail
Space O(1) One traversal pointer + one saved return value — no allocation

public int removeAnywhere(int position)
{

display() Time: O(n) Space: O(1) Traverse and print all elements
⚠️ Cannot use while(p != null) — p will never be null in a circular list. The loop uses a counter: while(i < length()).
Time O(n) Every node is visited exactly once — proportional to list size
Space O(1) Only a pointer p and counter i — no extra structures
Infinite loop risk High Without the counter, p != null never triggers — loop runs forever
csharp
public void display()
{
    Node p = head;
    int i = 0;
    while(i < length())      // counter-based — NOT while(p != null)!
    {
        Console.WriteLine(p.element + ", ");
        p = p.next;
        i++;
    }
    Console.WriteLine();
}

Q01 DEF What is the one structural property that makes a circular linked list different from a linear one?
Answer
The last node's next pointer points back to head instead of null. This creates a closed loop — there is no "end" to the list.
Example: In a list 7→4→12, the node holding 12 has its next field set to the memory address of node 7, not null.
Q02 GOTCHA Why can't you use while(p != null) to traverse a circular linked list?
Answer
Because p.next will never be null — the tail points back to head. The loop will keep going around the circle forever (infinite loop). You must use a counter: while(i < size).
Fix: int i = 0; while(i < length()) { ... p = p.next; i++; }
Q03 SEQUENCE Walk through the exact steps of addLast(e) when the list is NOT empty.
Answer
① Create newest node with element e and null next.
② newest.next = tail.next — since tail.next = head, newest now links to head.
③ tail.next = newest — old tail now links forward to newest.
④ tail = newest — tail pointer advances to the new node.
⑤ size++
Why step ② first? newest.next = tail.next must happen before tail.next is overwritten in step ③. Otherwise newest can't find the head.
Q04 APPLY A list contains: head→[7]→[4]→[12]←tail, and tail.next = head. Write the exact pointer assignments that happen when you call addFirst(23).
Answer
Node newest = new Node(23, null);
tail.next = newest; ← node 12 now links to node 23
newest.next = head; ← node 23 links to node 7 (old head)
head = newest; ← head pointer moves to node 23
size++;

Result: head→[23]→[7]→[4]→[12]→(back to 23)←tail
Critical order: tail.next = newest BEFORE head = newest. If you move head first, you can't set tail.next correctly.
Q05 GOTCHA In addFirst(), why must tail.next = newest happen BEFORE head = newest?
Answer
Once you write head = newest, the variable head now points to the new node. You lose the reference to the old head. But tail.next still needs to point to the new head (newest), not the old one. If you swap the order, tail.next = head would assign the new head to tail.next — which happens to be correct, BUT the issue is that newest.next = head would then point to itself. The safe, readable order that the code uses avoids all of this.
Q06 APPLY In addAnywhere(e, pos), what are the two pointer assignments made after traversal, and why does the ORDER matter?
Answer
① newest.next = p.next — new node links forward to what p used to link to.
② p.next = newest — p now links to the new node.

Order matters because: if you do ② first (p.next = newest), then p.next is already newest, so newest.next = p.next would make newest point to itself. You'd lose all nodes after the insertion point.
Rule: Always link the new node FORWARD first, then cut and rewire the previous link.
Q07 SEQUENCE List the steps of removeFirst() in a circular linked list. What extra step is required that isn't needed in a linear list?
Answer
① e = head.element — save value to return.
② tail.next = head.next — this is the extra circular step — tail must now skip the deleted head and point to the new head.
③ head = head.next — head advances.
④ size--
⑤ If size = 0, set head = null and tail = null.
Extra step explained: In a linear list, tail doesn't care about head. In a circular list, tail.next = head is the defining link — it must always be updated when head changes.
Q08 WHY Why is removeLast() O(n) even though we have a direct tail reference?
Answer
Because after deleting tail, the new tail is the node just before the deleted one — and singly-linked nodes have no previous pointer. You can't reach the second-to-last node from tail directly. The only way is to traverse from head until you reach the node at size - 1. That traversal is O(n).
Fix (if needed): A doubly-linked list would make this O(1) because each node also stores a prev pointer.
Q09 APPLY In removeLast(), after traversal, the code does: tail = p; p = p.next; tail.next = head;. What does each line do? Why must tail.next = head come BEFORE accessing p.element?
Answer
tail = p — the node we stopped at (second-to-last) becomes the new tail.
p = p.next — p now points to the old tail (the node being deleted).
tail.next = head — the new tail must wrap back to head to maintain circular property.

Order: tail.next = head must happen after tail = p so we know which node is the new tail. There's no issue with reading p.element after — p still points to the old tail node in memory (just no longer reachable from the list).
Q10 COMPARE How does a single-node circular linked list differ from a single-node linear linked list?
Answer
Linear: node.next = null. The list has a clear end.
Circular: node.next = node itself (self-loop). The list is already circular even with one node. head = tail = newest, and newest.next = newest.
In code (addLast empty case): newest.next = newest; head = newest; — the self-reference is what makes it circular from the start.

Q13 WHY Why is a circular linked list useful for modeling things like card games (poker, rummy)?
Answer
In a card game, turns cycle — after the last player, the first player goes again. A circular linked list models this naturally: traversing from any node and calling p = p.next forever will keep cycling through all players in order. There's no artificial "end" to manage. A linear list would need special-case logic ("if we're at the last player, wrap to first"), but in a circular list that wrapping is built into the structure itself.
Q14 GOTCHA What happens in removeFirst() if the list has exactly 1 element after deletion? What extra cleanup must happen?
Answer
After size--, if size becomes 0 (empty), both head and tail must be set to null. Without this cleanup, tail still points to the deleted node's memory address — leaving a dangling reference that could cause bugs or memory issues.
Code: if(IsEmpty()) { head = null; tail = null; }
Q15 COMPARE Which operations are O(1) vs O(n) in a circular linked list, and what determines which category an operation falls into?
Answer
O(1) operations: addFirst, addLast, removeFirst, length, isEmpty — because they only use existing head/tail pointers with no loops.

O(n) operations: addAnywhere, removeLast, removeAnywhere, display — because they require traversal (a while loop that runs proportional to the list size).

Q16 APPLY [Amazon] Detect whether a linked list has a cycle. Write the algorithm and explain its time and space complexity.
Answer
Use Floyd's Cycle Detection (Tortoise and Hare): two pointers — slow moves one step at a time, fast moves two steps. If they ever meet, there is a cycle. If fast reaches null, no cycle.
csharp
public bool HasCycle(Node head)
{
    Node slow = head, fast = head;
    while (fast != null && fast.next != null)
    {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) return true;   // cycle detected
    }
    return false;   // fast hit null → no cycle
}
Complexity: Time O(n) — fast catches slow within one full loop. Space O(1) — only two pointers, no extra data structure.
Q17 APPLY [Google] Given a circular linked list, find the starting node of the cycle (not just detect it). What algorithm do you use and why does it work?
Answer
Phase 1: Use Floyd's algorithm — let slow and fast meet inside the cycle.
Phase 2: Reset one pointer to head. Move both pointers one step at a time. They meet at the cycle start node.

Why it works: The math behind Floyd's shows that the distance from head to the cycle entry equals the distance from the meeting point to the cycle entry (going forward). So when both pointers advance at the same speed from their respective starting positions, they converge exactly at the entry.
csharp
// Phase 1: find meeting point
Node slow = head, fast = head;
while (fast != null && fast.next != null) {
    slow = slow.next; fast = fast.next.next;
    if (slow == fast) break;
}
// Phase 2: find cycle entry
slow = head;
while (slow != fast) { slow = slow.next; fast = fast.next; }
return slow;   // cycle start node
Complexity: Time O(n), Space O(1).
Q18 WHY [Microsoft] When would you choose a circular linked list over an array or a regular linked list? Give two concrete real-world use cases.
Answer
1. Round-Robin CPU Scheduling: The OS cycles through processes in a fixed order — after the last process, it wraps back to the first. A circular linked list models this naturally: p = p.next always gives the next process, with no end-check needed.

2. Multiplayer Board Games: Player turns cycle — after the last player, control returns to player 1. With a circular list, the "next player" logic is identical for every seat: just traverse forward.

Over arrays: Dynamic insertion/removal of players (processes) is O(1) vs O(n) for arrays.
Over linear lists: No special "wrap-around" condition needed — the structure handles it structurally.
Q19 APPLY [Flipkart] Write a function to find the length of the loop in a circular linked list using Floyd's algorithm. You may NOT use the size field.
Answer
After detecting the meeting point with Floyd's, keep one pointer fixed at the meeting point and move the other one step at a time, counting until it returns to the meeting point.
csharp
public int CycleLength(Node head)
{
    Node slow = head, fast = head;
    // Phase 1: find meeting point
    while (fast != null && fast.next != null) {
        slow = slow.next; fast = fast.next.next;
        if (slow == fast) {
            // Phase 2: count loop length
            int count = 1;
            Node current = slow.next;
            while (current != slow) { current = current.next; count++; }
            return count;
        }
    }
    return 0;   // no cycle
}
Complexity: Time O(n), Space O(1).

Q21 APPLY [Google] Split a circular linked list of even size into two equal circular linked lists. Describe the algorithm and the pointer assignments required.
Answer
Algorithm (slow/fast pointer to find midpoint):
① Use slow (1 step) and fast (2 steps) — when fast completes the loop, slow is at the midpoint.
② Head of list 2 = slow.next.
③ Break the first half's circle: fast.next = head (fast is at the last node; it must point back to head, not slow.next).
④ Break the second half's circle: slow.next = head2 (slow's next was already head2; now slow.next becomes the new circular wrap of list 2, which should already close correctly — verify fast.next == head2 to close list 2's circle).

Result: Two independent circular lists: list1 = head to slow, list2 = head2 back to itself.
Complexity: Time O(n) — one pass with slow/fast. Space O(1) — only pointer manipulation.
Q22 GOTCHA [Microsoft] A junior dev writes while(p.next != head) to traverse a circular linked list. What is the bug, and under what condition does it silently produce wrong output instead of crashing?
Answer
The bug: while(p.next != head) starts with p = head and stops one node early — it never processes the last node (whose next is head). The final element is silently skipped.

Condition for silent failure: It doesn't crash because the loop terminates normally — it just exits one iteration too soon. No exception is thrown. If you're summing elements or searching, the last node is invisible to the algorithm.

Fixes: Either use a counter while(i < size), or use a do-while loop: do { process(p); p = p.next; } while(p != head);
Classic interview trap: Off-by-one errors in circular structures are a favourite for interviewers — always trace through with a 1-node list and a 2-node list to verify boundary behaviour.
Q23 APPLY [Amazon] Implement the Josephus Problem: n people stand in a circle, every k-th person is eliminated. Find the position of the survivor. Describe how a circular linked list solves this naturally.
Answer
A circular linked list is the natural model — each node represents a person, node.next is the next person in the circle.

Algorithm: Start at head. Advance k−1 steps (so pointer p lands on the person before the one being eliminated). Remove p.next using p.next = p.next.next. Repeat until one node remains (it points to itself).
csharp
public int Josephus(int n, int k)
{
    // Build circular list 1..n
    Node head = new Node(1, null); head.next = head;
    Node tail = head;
    for (int i = 2; i <= n; i++) {
        tail.next = new Node(i, head);
        tail = tail.next;
    }
    Node p = tail;   // start one step before head
    while (p.next != p) {
        for (int i = 0; i < k - 1; i++) p = p.next;   // advance k-1
        p.next = p.next.next;                           // eliminate
    }
    return p.element;   // survivor
}
Complexity: Time O(n·k), Space O(n). The mathematical recurrence solution is O(n) time O(1) space — but the linked list version demonstrates the data structure directly.
Q24 SEQUENCE [Infosys / TCS] Explain the exact sequence of pointer changes needed to reverse a circular linked list in-place. What is the time and space complexity?
Answer
Strategy: Similar to reversing a linear linked list, but the final step must restore the circular link.

① Use three pointers: prev = tail (will become new tail, initially set to null for the reversal logic), current = head, next = null.
② Traverse the list with a counter. For each node: save next = current.next, reverse current.next = prev, advance prev = current, advance current = next.
③ After n iterations: prev is the new head. Set tail = head (old head is now the last node).
④ Restore circular link: tail.next = prev (new head). Set head = prev.

Result: All next pointers are reversed, head and tail are swapped, and the circular link is re-established.
Complexity: Time O(n) — single pass. Space O(1) — three pointers only, no extra memory.
Q25 WHY [Google / System Design] A music player needs a "loop all songs" feature. Would you implement the queue internally as an array, a linear linked list, or a circular linked list? Justify every choice with a specific complexity reason.
Answer
Circular linked list is the best fit. Here's the comparison:

Array: Loop-all requires a mod operation (index % n) — O(1) access, but inserting/removing songs mid-playlist is O(n) due to shifting. Fixed size unless resized.

Linear linked list: Insert/remove is O(1) with a pointer, but "loop all" requires an explicit check: if at last node, jump to head. This conditional must execute on every "next song" call — extra branching logic for a fundamental operation.

Circular linked list: "Next song" is always current = current.next — no branch, no mod. Looping is the default behaviour, not a special case. Adding/removing songs is O(1) at head/tail, O(n) mid-list. The structure directly models the domain.
Key insight: When the real-world problem has a cyclic structure (circular playlist, round-robin, turn-based games), a circular linked list eliminates conditional wrap-around logic — the data structure itself encodes the invariant.
