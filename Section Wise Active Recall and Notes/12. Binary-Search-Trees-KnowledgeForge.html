Binary Search Trees (BST)

Binary Search Trees

A sorted binary tree — every left child is smaller, every right child is larger. This one rule gives you O(log n) search, insert, and delete in the average case.

1. Definition & Properties

💡 Think of a BST like a sorted phone book — at every page you open, everything to the left is earlier in the alphabet, everything to the right is later.
  • A binary tree where every node satisfies the BST ordering rule
  • Left subtree → all values less than the node
  • Right subtree → all values greater than the node
  • This rule applies recursively to every node in the tree
  • No duplicates — each element is unique
  • In-order traversal always produces a sorted (ascending) sequence
  • All BSTs are binary trees, but not all binary trees are BSTs

BST Validation

📌 The BST property must hold for every single node, not just the root. Node 3's left child must be < 3, not just < 5.

2. Representation — Arrays vs. Linked Lists

💡 Imagine storing a family tree in a spreadsheet (array) vs. sticky notes with arrows (linked list). Sparse trees waste most spreadsheet cells but sticky notes only use what's needed.

Array Representation (Why It's Inefficient)

  • Root at index 1, left child of node at i → index 2i, right child → 2i+1
  • Problem: skewed trees waste massive memory — a tree of height h needs 2ʰ array slots
  • A sorted insertion (worst case) creates a chain — 7 elements need 128 slots, 120 wasted
  • No dynamic resizing — you must guess max size upfront

Linked List Representation (Preferred)

  • Each node = { element | left_ref | right_ref }
  • Memory allocated only for actual nodes — zero waste
  • Dynamic — grows and shrinks freely
⚠️ Array representation is only practical for complete binary trees (heaps). For general BSTs — always use linked nodes.

3. Search — The Core Concept

💡 Searching a BST is like the "hot/cold" game — at every node, you get a definitive hint: go left (smaller) or go right (larger). You never backtrack.
  • Start at the root
  • Compare target key with current node:
    • Match → return true
    • Key < node → recurse/move to left subtree
    • Key > node → recurse/move to right subtree
  • Reach null without match → return false

Why Can We Eliminate Half the Tree Each Step?

  • Because of the BST property: if target < current, every single node in the right subtree is also > current > target — so the whole right subtree can be skipped instantly
  • This is what makes BST search O(h) instead of O(n) — each comparison cuts the remaining search space roughly in half
  • In a balanced BST (height ≈ log n), this gives O(log n) comparisons — about 20 comparisons for 1 million nodes
📌 O(h) means the search never visits more than h+1 nodes — one per level of the tree. For a balanced BST of 7 nodes (height=3), that's at most 4 comparisons. For a skewed BST of 7 nodes (height=7), that's up to 8 comparisons.

4. Search Algorithms — Iterative vs. Recursive

✅ Iterative Search
bool SearchIterative(Node root, int key) {
    Node t = root;          // temp pointer
    while (t != null) {
        if (key == t.data)  return true;
        else if (key < t.data) t = t.left;
        else                   t = t.right;
    }
    return false;           // not found
}
↺ Recursive Search
bool SearchRecursive(Node root, int key) {
    if (root == null)      return false;  // base case
    if (key == root.data)  return true;   // found
    if (key < root.data)
        return SearchRecursive(root.left, key);
    else
        return SearchRecursive(root.right, key);
}

5. Insertion — Concept

💡 Insertion = search until you fall off the tree, then attach a new leaf exactly where you fell off.
  • Start at root, navigate exactly like a search
  • If key equals any existing node → reject (no duplicates)
  • When you reach a null slot → that's where the new node goes
  • New nodes are always inserted as leaf nodes

Why New Nodes Always Land as Leaves

  • The search navigates by comparing values — it can only stop at a node that's already there, or fall off the bottom of the tree (null)
  • Falling off at null means: that slot is exactly where this value belongs in the sorted order
  • Creating a new node there and linking it to the parent preserves the BST property automatically — no restructuring needed
  • This is why BST insertion is simpler than, say, AVL or Red-Black tree insertion — no rotations, just navigate and attach

What the Algorithm Tracks

  • Iterative: uses a trailing temp pointer to remember the parent (because when you hit null, you've lost the parent)
  • Recursive: the call stack itself remembers the path — the return value links the new node back up automatically

6. Insertion Algorithms — Iterative vs. Recursive

✅ Iterative Insert
void InsertIterative(ref Node root, int e) {
    Node temp = null, t = root;
    Node newNode = new Node(e);
    while (t != null) {
        temp = t;                       // track parent
        if (e == t.data) return;        // no duplicates
        else if (e < t.data) t = t.left;
        else                 t = t.right;
    }
    if (root == null) { root = newNode; return; }
    // attach to parent
    if (e < temp.data) temp.left  = newNode;
    else               temp.right = newNode;
}
↺ Recursive Insert
Node InsertRecursive(Node root, int e) {
    if (root == null)        // base: empty spot found
        return new Node(e);
    if (e == root.data)      // no duplicates
        return root;
    if (e < root.data)
        root.left  = InsertRecursive(root.left,  e);
    else
        root.right = InsertRecursive(root.right, e);
    return root;             // propagate reference up
}

📌 Recursive insert is elegant but builds O(h) stack frames. For large or skewed BSTs, iterative is the safe production choice.

7. BST Traversals — All Four Orders

💡 Think of traversal orders like reading a book: Pre-order = read the title first, In-order = read left page → bookmark → right page, Post-order = read every chapter before the summary, Level-order = read row by row left to right.

8. Traversal Algorithms — Recursive Code

In-Order (Left → Root → Right)

void InOrder(Node root) {
    if (root == null) return;         // base case: empty subtree
    InOrder(root.left);               // 1. recurse left
    Console.Write(root.data + " ");   // 2. visit root

Pre-Order (Root → Left → Right)

csharp
void PreOrder(Node root) {
    if (root == null) return;         // base case
    Console.Write(root.data + " ");   // 1. visit root FIRST
    PreOrder(root.left);              // 2. recurse left
    PreOrder(root.right);             // 3. recurse right
}
// Output: 50 30 10 40 80 60 90

Post-Order (Left → Right → Root)

csharp
void PostOrder(Node root) {
    if (root == null) return;         // base case
    PostOrder(root.left);             // 1. recurse left
    PostOrder(root.right);            // 2. recurse right
    Console.Write(root.data + " ");   // 3. visit root LAST
}
// Output: 10 40 30 60 90 80 50
⚠️ All three recursive traversals have the same structure — only the position of the Console.Write line changes! Pre = print first, In = print middle, Post = print last.

9. Level-Order Traversal — Queue-Based

💡 Think of level-order like a printer queue — you process each row before moving to the next. A queue handles this naturally: enqueue children as you dequeue parents.
csharp
void LevelOrder(Node root) {
    if (root == null) return;
    Queue<Node> q = new Queue<Node>();
    q.Enqueue(root);                   // start with root
    while (q.Count > 0) {
        Node curr = q.Dequeue();       // process front node
        Console.Write(curr.data + " ");
        if (curr.left  != null) q.Enqueue(curr.left);   // add left child
        if (curr.right != null) q.Enqueue(curr.right);  // add right child
    }
}
// Output for BST above: 50 30 80 10 40 60 90

10. BST Deletion — All Three Cases

💡 Deletion has 3 cases. Case 1 is trivial (snip a leaf). Case 2 is a bypass (bridge over the removed node). Case 3 is a swap-and-delete (replace with the closest value, then delete that node).

The Reference BST for All Deletion Examples

Case 1 — Leaf Node (No Children)

  • Find the target node and its parent
  • Set parent's left or right pointer to null
  • Free/discard the node
  • Why it's simple: A leaf has no children, so removing it disconnects nothing downstream. The parent just stops pointing at it — the tree heals itself instantly.
💡 Deleting 15: parent is 12. Check: is 15 the left or right child of 12? → right. Set 12.right = null. Done. Node 15 is gone.

Case 2 — One Child

  • Find target node (n) and its parent (p)
  • Set parent's pointer to target's only child — bridges over the gap
  • Delete target node
  • Why it works: The target has one child subtree. That entire subtree already satisfies BST property (it was valid before). Connecting it directly to the grandparent skips the deleted node without breaking any ordering rule.
  • Either child works: Whether it's a left-only or right-only child, the bypass is the same — just redirect the parent's pointer to skip the deleted node
💡 Deleting 12 (has only right child 15): parent is 40. 12 is 40's left child. Set 40.left = 12.right (15). Node 40 now directly points to 15. Node 12 is disconnected.

Case 3 — Two Children (Most Complex)

  • Cannot simply redirect parent — parent can only hold one reference, but target has two subtrees
  • Strategy: Don't actually remove the node — instead replace its value with the closest valid replacement, then delete that replacement node
  • Two valid replacements:
    • In-order predecessor (largest in left subtree) — go left once, then right as far as possible
    • In-order successor (smallest in right subtree) — go right once, then left as far as possible
  • Why the predecessor/successor works: The predecessor is guaranteed to be > everything in the left subtree AND < everything in the right subtree — copying it up maintains BST ordering at every node
  • Why the replacement itself is easy to delete: The in-order predecessor has NO right child (if it did, that right child would be larger — contradicting "it's the rightmost"). So it's either a leaf (Case 1) or has only a left child (Case 2) — never Case 3 again
📌 The replacement (in-order successor or predecessor) is guaranteed to have at most ONE child — so its deletion is always Case 1 or Case 2, never recursive Case 3.
⚠️ The code in Program.cs uses the in-order predecessor (largest in left subtree). Some implementations use the in-order successor (smallest in right subtree). Both are equally valid — choose one and be consistent.
csharp
Node Delete(Node root, int key) {
    if (root == null) return null;         // not found

    // Step 1: Navigate to the target node
    if      (key < root.data) root.left  = Delete(root.left,  key);
    else if (key > root.data) root.right = Delete(root.right, key);
    else {
        // Found the target node — handle 3 cases:

        // Case 1 & 2: zero or one child
        if (root.left  == null) return root.right;   // promote right child (or null)
        if (root.right == null) return root.left;    // promote left child

        // Case 3: two children
        // Find in-order successor (smallest in right subtree)
        Node successor = FindMin(root.right);        // Step 2: find min of right
        root.data = successor.data;                  // Step 3: copy successor's value
        root.right = Delete(root.right, successor.data); // Step 4: delete successor
    }
    return root;
}

Node FindMin(Node node) {
    while (node.left != null) node = node.left;  // go as far left as possible
    return node;
}

11. Count & Height — Recursive Algorithms

💡 Both Count and Height follow the same pattern: solve the problem for left and right subtrees first (recursively), then combine the answers at the current node. This is called divide and conquer.

How the Recursion Pattern Works

  • Base case: null node → return 0. This is the "ground floor" that stops infinite recursion
  • Recursive case: Compute for left, compute for right, combine + 1 (for current node)
  • The call stack unwinds from leaves upward — leaf nodes return 1, their parents add their subtrees together + 1, and so on up to root
  • Think of it as: the tree reports its own size/height from the bottom up, like a chain of messengers passing numbers up
✅ Count Nodes
int Count(Node root) {
    if (root == null) return 0;
    int left  = Count(root.left);
    int right = Count(root.right);
    return left + right + 1;  // +1 for root
}
// BST {50,30,80,10,40,60,90} → 7
↕ Height of Tree
int Height(Node root) {
    if (root == null) return 0;
    int left  = Height(root.left);
    int right = Height(root.right);
    return Math.Max(left, right) + 1;
}
// Same BST → height = 3 (3 levels)

Count Trace on BST {50, 30, 80, 10, 40, 60, 90}

📌 Both Count and Height are O(n) time — they must visit every node. Space: O(h) for the recursive call stack.

Node Class — The Building Block

💡 Every piece of data in a BST lives inside a Node. Think of it as a box that holds a number and two arrows — one pointing left, one pointing right.

class Node

Insert() — Iterative Insertion

💡 This is the iterative insert. It walks down the tree keeping a "parent" pointer (temp), then attaches the new node to the correct side of the parent once it finds the empty slot.
csharp
public void Insert(Node? tempRoot, int e)
{
    Node? temp = null;          // will track the PARENT of insertion point
    while (tempRoot != null)    // walk down until we fall off the tree
    {
        temp = tempRoot;        // save current node as potential parent
        if (e == tempRoot.Element)
            return;             // duplicate — reject, don't insert
        else if (e > tempRoot.Element)
            tempRoot = tempRoot.Right;  // go right (larger values)
        else if (e < tempRoot.Element)
            tempRoot = tempRoot.Left;   // go left (smaller values)
    }
    // tempRoot is now null — we found the insertion slot
    Node n = new Node(e, null, null);   // create the new leaf node

    if (Root != null && temp != null)   // tree was NOT empty
    {
        if (e < temp.Element)
            temp.Left = n;    // attach as left child
        else
            temp.Right = n;   // attach as right child
    }
    else
    {
        Root = n;             // tree was empty — this node becomes root
    }
}
  • Why two pointers? — tempRoot moves forward; temp trails behind. When tempRoot becomes null, temp is the parent you need to attach to
  • Duplicate check — if (e == tempRoot.Element) return; — BSTs reject duplicates entirely
  • Empty tree case — if Root is null, the new node simply becomes the root
  • Time: O(h)  |  Space: O(1) — no recursion stack
⚠️ Notice: Insert(B.Root, 50) is called with B.Root (which is null initially). The first call sets Root = n making 50 the root. Subsequent calls pass the actual root, and the while loop finds the right slot.

InsertRecursion() — Recursive Insertion

💡 The recursive version is shorter and more elegant. It returns the (possibly new) root at each level, and the caller automatically links it back: tempRoot.Left = InsertRecursion(tempRoot.Left, e).
csharp
public Node InsertRecursion(Node tempRoot, int e)
{
    if (tempRoot != null)           // still navigating — not at insertion point yet
    {
        if (e < tempRoot.Element)
            tempRoot.Left = InsertRecursion(tempRoot.Left, e);  // go left, link result back
        else if (e > tempRoot.Element)
            tempRoot.Right = InsertRecursion(tempRoot.Right, e); // go right, link result back
        // if e == tempRoot.Element: neither branch runs → implicit duplicate rejection
    }
    else
    {
        // tempRoot is null = we found the empty slot — create and return the new node
        Node n = new Node(e, null, null);
        tempRoot = n;
    }
    return tempRoot;   // return the root of this subtree (unchanged or newly created)
}

// USAGE — must assign root back on first call:
// B2.Root = B2.InsertRecursion(B2.Root, 70);
  • Key insight: tempRoot.Left = InsertRecursion(tempRoot.Left, e) — this line both recurses AND links the returned node back, so the tree stays connected automatically
  • Why return Node? — Because the base case creates a new node that must be linked to the tree. Returning it propagates the reference up the call stack
  • Critical usage note: The very first call MUST capture the return value: B2.Root = B2.InsertRecursion(B2.Root, 70); — subsequent calls don't need this because the existing root is returned unchanged
  • Time: O(h)  |  Space: O(h) call stack frames

InOrderTraversal() — Left → Root → Right

csharp
public void InOrderTraversal(Node tempRoot)
{
    if (tempRoot != null)              // base case: empty subtree → do nothing
    {
        InOrderTraversal(tempRoot.Left);              // 1. recurse LEFT
        Console.Write(tempRoot.Element + ", ");       // 2. VISIT (print)
        InOrderTraversal(tempRoot.Right);             // 3. recurse RIGHT

}
// BST {50,30,80,10,40,60,90} → Output: 10, 30, 40, 50, 60, 80, 90 (sorted!)
  • The if (tempRoot != null) guard stops recursion at leaf boundaries — no null check needed before each call because the guard handles it
  • For a BST specifically, in-order always produces ascending sorted output
  • Time: O(n) — visits every node exactly once

LevelOrderTraversal() — BFS Using Queue

💡 Level-order prints nodes row by row. The Queue is the key: enqueue the root, then in each loop iteration — dequeue a node, print it, enqueue its children. Children of each level get processed before grandchildren.
csharp
public void LevelOrderTraversal()
{
    Queue<Node> q = new Queue<Node>();
    Node t = Root;
    Console.Write(t.Element + ", ");   // print root first
    q.Enqueue(t);                      // enqueue root to start the loop
    while (q.Count != 0)
    {
        t = q.Dequeue();               // take the front node
        if (t.Left != null)            // if left child exists:
        {
            Console.Write(t.Left.Element + ", ");  // print it
            q.Enqueue(t.Left);                     // and queue it for later
        }
        if (t.Right != null)           // if right child exists:
        {
            Console.Write(t.Right.Element + ", "); // print it
            q.Enqueue(t.Right);                    // and queue it for later
        }
    }
}
// Output: 50, 30, 80, 10, 40, 60, 90
  • Why Queue (FIFO)? — First in, first out guarantees all nodes of level N are processed before any node of level N+1
  • Note: This implementation prints root before the loop starts, then prints children inside the loop — so print happens before enqueue. An alternative is to only print inside the loop after dequeue (both are valid)
  • Time: O(n)  |  Space: O(w) where w = maximum width of the tree

ContainsIterative() — Iterative Search

csharp
public bool ContainsIterative(int key)
{
    Node temptRoot = Root;           // start from root (doesn't modify Root)
    while (temptRoot != null)        // loop until we fall off the tree
    {
        if (temptRoot.Element == key)
            return true;             // found it!
        else if (key < temptRoot.Element)
            temptRoot = temptRoot.Left;   // key is smaller → go left
        else if (key > temptRoot.Element)
            temptRoot = temptRoot.Right;  // key is larger → go right
    }
    return false;                    // exited loop without finding key
}
// B.ContainsIterative(50) → true
// B.ContainsIterative(100) → false
  • Uses a local copy of Root (temptRoot) — never modifies the actual tree pointer
  • Three outcomes per node: found, go left, go right. Only stops when found or hits null
  • Time: O(h)  |  Space: O(1) — single pointer, no stack

ContainsRecursive() — Recursive Search

csharp
public bool ContainsRecursive(Node tempRoot, int Key)
{
    if (tempRoot != null)
    {
        if (tempRoot.Element == Key)
            return true;                              // base case: found
        else if (Key < tempRoot.Element)
            return ContainsRecursive(tempRoot.Left, Key);   // search left subtree
        else if (Key > tempRoot.Element)
            return ContainsRecursive(tempRoot.Right, Key);  // search right subtree
    }
    return false;   // tempRoot is null → key doesn't exist
}
// B.ContainsRecursive(B.Root, 50) → true
  • Each recursive call narrows the search to one subtree — never both
  • The function returns false both when tempRoot is null (not found) AND when none of the if-branches match (which only happens if tempRoot is null)
  • Time: O(h)  |  Space: O(h) — one stack frame per level traversed

Delete() — The Complete Deletion Algorithm

💡 This is the most complex method in the file. It handles all 3 deletion cases in one function. Read the inline comments — they explain each step and WHY it's done that way.
csharp
public bool Delete(int e)
{
    // ── STEP 1: Find the node to delete ──────────────────────────────────────
    // Use two pointers: 'current' is the node being examined,
    // 'parent' is its parent — we need parent to re-link after deletion.
    Node current = Root;
    Node parent = null;

    while (current != null && current.Element != e)
    {
        parent = current;                   // save current as parent before moving
        if (e < current.Element)
            current = current.Left;         // target is smaller → go left
        else
            current = current.Right;        // target is larger → go right
    }

    // ── STEP 2: Not found ────────────────────────────────────────────────────
    if (current == null)
    {
        Console.WriteLine("Element not found");
        return false;
    }

    // ── STEP 3: Two children (Case 3 — most complex) ─────────────────────────
    // Problem: We can't simply remove current because it has two subtrees
    // and the parent can only hold ONE reference.
    //
    // Solution: Find the in-order PREDECESSOR (largest in LEFT subtree).
    // Copy its value into 'current'. Then delete the predecessor node instead
    // (which has at most one child — making it Case 1 or Case 2).
    //
    // Why predecessor? Because it's smaller than everything in the right subtree
    // and larger than everything else in the left subtree — so BST property holds.
    if (current.Left != null && current.Right != null)
    {
        Node predecessor = current.Left;        // start at left child
        Node predecessorParent = current;

        while (predecessor.Right != null)       // go as far RIGHT as possible
        {                                       // (rightmost of left subtree = largest)
            predecessorParent = predecessor;
            predecessor = predecessor.Right;
        }

        current.Element = predecessor.Element;  // overwrite current's value with predecessor's
        // Now 'current' is logically deleted — we just need to remove 'predecessor'
        current = predecessor;                  // redirect deletion to predecessor node
        parent = predecessorParent;
    }

    // ── STEP 4: Find the surviving child (Case 1 or Case 2) ─────────────────
    // After Step 3, 'current' has AT MOST one child.
    // Case 1 (leaf): child = null
    // Case 2a (left child only): child = current.Left
    // Case 2b (right child only): child = current.Right
    Node child = null;
    if (current.Left != null)
        child = current.Left;
    else
        child = current.Right;      // also null if it's a leaf — that's fine

    // ── STEP 5: Re-link the parent ───────────────────────────────────────────
    if (current == Root)
        Root = child;               // deleting root → its only child becomes new root
    else
    {
        if (current == parent.Left)
            parent.Left = child;    // current was left child → replace with child
        else
            parent.Right = child;   // current was right child → replace with child
    }

    return true;
}
// B.Delete(90) removes 90 (leaf — Case 1). parent.Right (80.Right) = null.

Step-by-Step: What Delete(90) Does

  • Step 1 Search: current=50, e=90>50 → right; current=80, e=90>80 → right; current=90 ✓ found. parent=80
  • Step 3 check: 90.Left=null → not two children → skip Case 3
  • Step 4: 90.Left=null → child = 90.Right = null (leaf)
  • Step 5: 90 ≠ Root. 90 == parent(80).Right → 80.Right = null
  • Result: Node 90 is detached. Tree now has 80 as a right child of 50 with no right child.

Count() — Count All Nodes Recursively

csharp
public int Count(Node tempRoot)
{
    if (tempRoot != null)
    {
        int x = Count(tempRoot.Left);   // count all nodes in left subtree
        int y = Count(tempRoot.Right);  // count all nodes in right subtree
        return x + y + 1;              // +1 = the current node itself
    }
    return 0;   // null subtree contributes 0
}
// For BST {50,30,80,10,40,60} (after deleting 90): returns 6
  • Algorithm logic: Total nodes at any subtree = (left subtree count) + (right subtree count) + 1 (for the root of this subtree)
  • Leaf nodes: both subtrees are null → Count(null)=0 → returns 0+0+1 = 1
  • Empty tree: Count(null) returns 0
  • Time: O(n) — must visit every node  |  Space: O(h) recursion stack

Height() — Tree Height Recursively

csharp
public int Height(Node tempRoot)
{
    if (tempRoot != null)
    {
        int x = Height(tempRoot.Left);   // height of left subtree
        int y = Height(tempRoot.Right);  // height of right subtree
        if (x > y)
            return x + 1;   // left side is taller — use it, add 1 for this level
        else
            return y + 1;   // right side is taller (or equal) — use it, add 1
    }
    return 0;   // null node has height 0
}
// BST {50,30,80,10,40,60} → 3
// Equivalent to: return Math.Max(Height(left), Height(right)) + 1
  • Algorithm logic: The height of a subtree = 1 (for the current node) + max(left height, right height)
  • Leaf nodes: both children null → returns max(0,0)+1 = 1
  • Empty tree: returns 0
  • The if/else in this code does the same thing as Math.Max(x, y) + 1
  • Time: O(n) — visits every node  |  Space: O(h) recursion stack

📘 Foundations & Concepts (Q01–Q20)
Q01DEF What is a Binary Search Tree? State the core ordering rule.
Answer
A BST is a binary tree where every node's left subtree contains only values smaller than the node, and its right subtree contains only values larger. This rule applies recursively to every node, not just the root.
Example: Root=5, left child=3 (valid, 3<5), right child=8 (valid, 8>5), left grandchild=6 (invalid — 6>5 so it can't be under 3).
Q02WHY Why is in-order traversal of a BST guaranteed to produce sorted output?
Answer
Because the BST property ensures left < current < right at every node. In-order visits left subtree first (all smaller), then current node, then right subtree (all larger). This naturally visits nodes in ascending order.
Example: BST {50,30,80,10,40,60,90} — in-order: 10→30→40→50→60→80→90. Sorted!
Q03GOTCHA If node 3 has a right child of 5, and node 3 is the left child of node 6 — is this a valid BST? Why or why not?
Answer
Invalid. Even though 5 > 3 (locally valid), node 5 is in the left subtree of 6, which requires all values to be < 6. The BST property must hold globally, not just locally between parent and child.
Key insight: Always check the global constraint — a node's value must fit within the entire ancestor chain's bounds, not just its immediate parent.
Q04WHY Why is array representation inefficient for BSTs? Why do we use linked nodes instead?
Answer
Arrays require 2ʰ slots for a tree of height h. A skewed BST of 7 elements can have height 6, needing 128 slots — 121 wasted. Linked nodes only allocate memory for actual nodes: each is { data | left_ref | right_ref }, zero waste, dynamic sizing.
Rule: Array BST = only practical for complete/perfect binary trees (heaps). General BSTs always use linked representation.
Q05SEQUENCE List the steps of BST search in order. What happens when the element is not found?
Answer
① Start at root. ② Compare key with current node. ③ If match → return true. ④ If key < current → move left. ⑤ If key > current → move right. ⑥ Repeat until found or reach null. ⑦ If null reached → return false (not found).
Example: Search 7 in {5,3,8,1,4,6,9}: 7>5→R(8), 7<8→L(6), 7>6→R(null) → return false.
Q06COMPARE Iterative vs. Recursive BST search — what is the key difference in space complexity and which is preferred?
Answer
Iterative: O(1) space — uses only a pointer variable. Recursive: O(h) space — one call-stack frame per tree level. Both are O(h) time. Iterative preferred in production — no stack overflow on deep/skewed trees.
Gotcha: For a skewed BST of 1000 nodes, recursive search makes 1000 nested calls — stack overflow risk!
Q07APPLY Write the iterative BST search function in C#. What does the temp variable track in insert?
Answer
bool Search(Node root, int key) {
    Node t = root;
    while (t != null) {
        if (key == t.data) return true;
        else if (key < t.data) t = t.left;
        else t = t.right;
    }
    return false;
}
In insert, temp tracks the last non-null node visited — it becomes the parent for the new node.
Q08APPLY Where does a new node always land after BST insertion? What position in the tree?
Answer
A new node is always inserted as a leaf — at the position where the search falls off the tree (reaches null). The BST property guides navigation; insertion happens where the null slot is found.
Example: Insert 35 into {50,30,80,10,40,60,90}: 35<50→L, 35>30→R, 35<40→L(null). 35 becomes left child of 40.
Q09DEF Define the four BST traversals and state the visit order pattern for each.
Answer
In-Order: Left → Root → Right (produces sorted output in BST)
Pre-Order: Root → Left → Right (root visited first)
Post-Order: Left → Right → Root (root visited last)
Level-Order: Row by row top-to-bottom, left-to-right (uses queue)
Memory trick: "Pre" = root is first. "In" = root is in the middle. "Post" = root is last (postponed).
Q10GOTCHA What's the only structural difference between the InOrder, PreOrder, and PostOrder recursive functions?
Answer
Only the position of the print statement changes. Pre = print before recursing left. In = print between left and right recursion. Post = print after both recursions. All three functions have identical structure otherwise.
🔧 Operations (Q11–Q20)
Q11SEQUENCE List the three cases of BST deletion in order of complexity. What makes Case 3 the hardest?
Answer
Case 1 — Leaf: No children. Just nullify parent's pointer.
Case 2 — One child: Bypass the node — parent points directly to its only grandchild.
Case 3 — Two children: Hardest. Can't bypass because parent can only hold one pointer. Must find a replacement value (in-order successor or predecessor), swap, then delete the replacement.
Q12APPLY In Case 3 deletion, what is the in-order successor and why is it the right replacement?
Answer
The in-order successor is the smallest value in the right subtree (found by going right once, then left as far as possible). It's the right replacement because: (a) it's greater than everything in the left subtree, (b) it's smaller than everything else in the right subtree. Both BST properties are preserved after swap.
Key fact: The in-order successor has at most one child (no left child), so its own deletion is always Case 1 or Case 2 — never recursive Case 3.
Q13APPLY Write the recursive Count() function for a BST. What does it return for an empty tree?
Answer
int Count(Node root) {
    if (root == null) return 0;
    return Count(root.left) + Count(root.right) + 1;
}
Returns 0 for empty tree/subtree. Each node adds 1 to the sum of left and right subtree counts.
Q14APPLY Write Height() for a BST. What value does a leaf node return?
Answer
int Height(Node root) {
    if (root == null) return 0;
    return Math.Max(Height(root.left), Height(root.right)) + 1;
}
A leaf node: both children are null → Height(null)=0 both sides → max(0,0)+1 = 1.
Q15WHY Why does Level-Order traversal require a Queue but In/Pre/Post-Order don't need one explicitly?
Answer
In/Pre/Post-order use recursion, which implicitly uses the call stack for LIFO ordering — they process depth-first (go deep before going wide). Level-order needs FIFO ordering to process all nodes on one level before the next — a Queue provides FIFO explicitly.
Q16DRAW Draw the BST formed by inserting: 50, 30, 70, 20, 40, 60, 80. Label all nodes.
Answer
Check your labels: 50 at root, 30 and 70 at level 2, 20/40 under 30, 60/80 under 70. Height = 3. In-order: 20→30→40→50→60→70→80.
Q17COMPARE Iterative vs. Recursive BST insert — which is safer for production? Compare space complexity.
Answer
Both O(h) time. Iterative: O(1) space — just pointer variables. Recursive: O(h) space — one stack frame per level. Iterative is safer in production: no stack overflow risk on tall/skewed trees. Recursive is cleaner to read and explain in interviews.
Q18APPLY Given BST {50,30,80,10,40,60,90}, write the level-order traversal output step by step using a queue.
Answer
Queue starts: [50]. Dequeue 50 → print 50, enqueue 30, 80. Queue: [30,80]. Dequeue 30 → print 30, enqueue 10,40. Queue: [80,10,40]. Dequeue 80 → print 80, enqueue 60,90. Queue: [10,40,60,90]. Dequeue 10→40→60→90 (all leaves, no children enqueued).
Output: 50 → 30 → 80 → 10 → 40 → 60 → 90
Q19WHY Why does a BST not allow duplicate values? What happens if you try to insert a duplicate?
Answer
BSTs use unique keys for unambiguous lookup. If duplicates were allowed, search couldn't determine which copy was the target, and deletion would be unclear (delete all? the first one?). By convention, inserting a duplicate is rejected — the function returns without modifying the tree.
Q20SEQUENCE Walk through deleting node 60 from BST {50,30,80,10,40,60,90} step by step. Which deletion case is this?
Answer
Case 1 — Leaf deletion. ① Search for 60: 60>50→R(80), 60<80→L(60). ② Node 60 found. ③ Check children: 60 has no left, no right children (it's a leaf). ④ Set parent (80)'s left pointer to null. ⑤ Delete node 60. Tree now has 80 with only right child 90.
🎯 Real Interview Questions (Q21–Q50)
Q21APPLY What is the time complexity of BST operations in the worst case and when does worst case occur?
Answer
Worst case for all operations (search, insert, delete) is O(n). This occurs when the BST degenerates into a linked list — caused by inserting elements in sorted (or reverse-sorted) order. Every node has only one child, creating height = n instead of log n.
Example: Inserting 1,2,3,4,5,6,7 in order creates a right-skewed tree. Searching for 7 requires 7 comparisons = O(n).
Q22APPLY How do you find the minimum and maximum element in a BST? Write the algorithm.
Answer
Minimum: Go left as far as possible — the leftmost node. Maximum: Go right as far as possible — the rightmost node. 
Node FindMin(Node r) { while(r.left!=null) r=r.left; return r; }
Node FindMax(Node r) { while(r.right!=null) r=r.right; return r; }
Time: O(h). Space: O(1) iterative.
Q23COMPARE What's the difference between a BST and a Binary Heap? When would you use each?
Answer
BST: Ordered (left<node<right), supports O(log n) search/insert/delete. Arbitrary shape. Use for: dictionaries, sorted data, in-order output.
Heap: Parent ≥ or ≤ children (min/max-heap), no ordering between siblings. Complete binary tree. Use for: priority queues, finding min/max in O(1), heap sort. Heap can't search efficiently — O(n).
Q24APPLY How do you check if a binary tree is a valid BST? What's the naive approach's flaw?
Answer
Naive flaw: Checking only left<node and right>node locally misses global violations (a node in the wrong subtree of an ancestor).
Correct approach: Pass min/max bounds down recursively. Each node must be within (min, max). Left recursion tightens max; right recursion tightens min. 
bool IsValid(Node n, int min=int.MinValue, int max=int.MaxValue) {
    if (n==null) return true;
    if (n.data<=min || n.data>=max) return false;
    return IsValid(n.left,min,n.data) && IsValid(n.right,n.data,max);
}
Q25APPLY How do you find the kth smallest element in a BST? What traversal do you use?
Answer
Use in-order traversal — it visits nodes in sorted ascending order. Count nodes during traversal; when count reaches k, return the current node. Time: O(h+k). Space: O(h) for recursion stack.
Example: BST {10,30,40,50,60,80,90}, k=3 → in-order: 10,30,40. Answer: 40.
Q26WHY What problem does a self-balancing BST (like AVL or Red-Black tree) solve? What's the guarantee?
Answer
Regular BST degrades to O(n) on sorted/skewed input. Self-balancing BSTs maintain height ≈ log n through automatic rotations after insert/delete. AVL: Height difference between subtrees ≤ 1 (strict). Red-Black: Looser balance, fewer rotations, faster insert/delete in practice. Both guarantee O(log n) worst case for all operations.
Q27APPLY How do you find the Lowest Common Ancestor (LCA) of two nodes in a BST?
Answer
Exploit the BST property: start at root. If both nodes are smaller → LCA is in left subtree. If both larger → right subtree. If one is on each side (or equals current node) → current node is the LCA. O(h) time, O(1) space iteratively.
Example: LCA(10,40) in {50,30,80,10,40,60,90}: 10<50 and 40<50 → go left. At 30: 10<30 and 40>30 → LCA = 30.
Q28APPLY How do you convert a sorted array into a balanced BST? What's the key insight?
Answer
Pick the middle element as root — this ensures roughly equal elements on each side. Recursively do the same for left half (left subtree) and right half (right subtree). Time: O(n), Space: O(log n) recursion stack.
Example: [1,2,3,4,5,6,7] → mid=4(root), left=[1,2,3] mid=2, right=[5,6,7] mid=6. Result: perfectly balanced BST of height 3.
Q29APPLY How do you find the in-order successor of a given node in a BST (without parent pointer)?
Answer
Case A: Node has right child → successor = leftmost node of right subtree (FindMin of right subtree).
Case B: No right child → traverse from root; last node for which you went left is the successor. Track ancestor during descent.
Example: In-order successor of 30 in {50,30,80,10,40,60,90}: 30 has right child (40) → go left in 40 → no left → successor is 40.
Q30GOTCHA You search for element 7 in a BST. The search reaches a node with value 6 whose right child is null. What do you return and why?
Answer
Return false. Since 7>6, you'd go right — but right is null. Reaching null means the element doesn't exist in the BST. The search terminates without finding the key. This is the "not found" base case.
Q31APPLY How would you find all values in a BST within a given range [low, high]?
Answer
Use in-order traversal with pruning: if current node < low → skip left subtree (all smaller). If current node > high → skip right subtree (all larger). If within range → print and recurse both sides. Time: O(h + k) where k = number of results.
Q32COMPARE How does BST search compare to binary search on a sorted array? When would you prefer each?
Answer
Sorted array binary search: O(log n) guaranteed, O(1) space, but insert/delete are O(n) (shifting). BST search: O(log n) average (O(n) worst), insert/delete are O(log n) average. Use array when data is static; use BST when dynamic insert/delete is needed.
Q33APPLY How do you count the number of nodes in a BST with value in range [lo, hi]?
Answer
Recursive approach: if node == null → 0. If node.data < lo → only check right subtree. If node.data > hi → only check left subtree. If lo ≤ node.data ≤ hi → 1 + count(left) + count(right).
Q34WHY Pre-order traversal is called "useful for serializing a BST." Explain why.
Answer
Pre-order visits root first, then left subtree, then right subtree. If you re-insert values from a pre-order sequence into a new empty BST in that order, you reconstruct the exact same tree structure. In-order doesn't preserve structure (it only gives sorted values, which could come from any BST shape).
Q35APPLY How do you detect if a BST is height-balanced (all subtree heights differ by ≤ 1)?
Answer
Compute height recursively. If at any node |height(left) - height(right)| > 1, return unbalanced. Optimize by returning -1 as a sentinel for "unbalanced" to avoid recomputation. This runs O(n) by computing heights bottom-up in a single pass.
Q36GOTCHA You call recursive BST search on a skewed tree with 10,000 nodes. What runtime error might occur?
Answer
StackOverflowException. A skewed BST of 10,000 nodes has height = 10,000. Recursive search would make 10,000 nested calls, each pushing a stack frame. The default call stack is typically 1–8MB, which overflows around 5,000–10,000 frames depending on frame size. Solution: use iterative search.
Q37APPLY How do you merge two BSTs into one valid BST?
Answer
① In-order traverse both BSTs → two sorted arrays. ② Merge the two sorted arrays (like merge sort merge). ③ Convert the merged sorted array to a balanced BST (mid-element as root). Time: O(m+n), Space: O(m+n).
Q38APPLY How do you print all root-to-leaf paths in a BST?
Answer
Use DFS with a path list. At each node, add to path. When a leaf is reached (both children null), print the path. On returning from recursion, remove the current node from path (backtrack). Time: O(n·h) in worst case (n nodes, h path length to print).
Q39WHY Post-order traversal is useful for tree deletion. Explain why.
Answer
Post-order visits left subtree, right subtree, then root — children before parent. When freeing/deleting a tree, you must delete children before deleting the parent (otherwise you lose references to children). Post-order naturally ensures no node is deleted before its dependents are cleared.
Q40APPLY How do you find the height of a BST iteratively (using level-order)?
Answer
Use level-order traversal. Count how many levels (complete queue-draining iterations) are needed. Each "wave" of dequeuing represents one level. When the queue empties after k waves, height = k. Time: O(n), Space: O(w) where w = maximum width.
Q41COMPARE BST average case O(log n) vs. worst case O(n) — when does average case hold?
Answer
Average case O(log n) holds for random insertion order — mathematically proven that randomly ordered insertions produce expected height ≈ 2 log n. If insertion order is sorted (or reverse-sorted), worst case O(n) occurs immediately. "Average" assumes random data distribution.
Q42APPLY How do you check if two BSTs are identical (same structure AND same values)?
Answer
Recursive: if both null → true. If one null → false. If data differs → false. Return isIdentical(t1.left,t2.left) AND isIdentical(t1.right,t2.right). Must check structure AND values simultaneously. Time: O(n) where n = min of both tree sizes.
Q43APPLY Describe how to find the floor and ceiling of a value k in a BST (without inserting k).
Answer
Floor (largest value ≤ k): If node == k → floor is node. If k < node → floor is in left subtree. If k > node → floor might be node itself or in right subtree; recurse right, update floor if result is null.
Ceiling (smallest value ≥ k): Mirror logic. Time: O(h).
Q44GOTCHA Is the in-order traversal sequence unique to a BST? Can two different BSTs produce the same in-order output?
Answer
Yes — all BSTs containing the same set of values produce the same in-order output (the sorted sequence). A different BST with the same values would have different structure but identical in-order traversal. In-order alone cannot reconstruct the original BST structure.
Gotcha: To uniquely reconstruct a BST, you need pre-order + in-order, or post-order + in-order — not in-order alone.
Q45APPLY How would you implement BST deletion iteratively (not recursively)?
Answer
① Use two pointers: curr (target) and parent. Traverse to find target while tracking parent. ② For Case 1 (leaf): set parent.left or parent.right to null. ③ For Case 2 (one child): set parent's pointer to target's non-null child. ④ For Case 3: iteratively find in-order successor (go right once, then leftmost), copy successor's data into target node, then delete successor node (which is Case 1 or 2).
Q46COMPARE DFS vs BFS on a BST — what does each achieve? When would you use BFS over DFS?
Answer
DFS (in/pre/post-order): Explores depth-first, good for: searching, BST property verification, path problems, sorted output. Uses O(h) stack space.
BFS (level-order): Explores width-first, good for: finding shortest path, level-by-level processing, finding nodes close to root. Uses O(w) queue space where w = max width. Use BFS when proximity to root matters.
Q47APPLY How do you find the diameter (longest path between any two nodes) of a BST?
Answer
At each node, diameter through that node = height(left) + height(right). Recursively compute height of subtrees and update a global max at each node. Final answer = max over all nodes. Time: O(n). The diameter may or may not pass through the root.
Q48WHY Why is BST deletion more complex than insertion? What structural guarantee does insertion rely on?
Answer
Insertion always adds a leaf — it never restructures existing connections. Deletion may remove any node including internal nodes with two children, requiring subtree restructuring to maintain BST property. Case 3 deletion requires finding a replacement value and performing a secondary deletion, making it significantly more complex than always-appending insertion.
Q49APPLY Given in-order traversal [1,2,3,4,5] and pre-order [3,1,2,4,5], reconstruct the BST structure.
Answer
Pre-order first element = root = 3. In in-order, 3 splits [1,2] (left subtree) and [4,5] (right subtree). Pre-order next for left: 1 → root of left subtree. In-order [1,2]: 1 has no left, right=2. Pre-order next for right: 4 → root of right. In-order [4,5]: 4 has no left, right=5. Result: root=3, left=1(right=2), right=4(right=5).
Q50SEQUENCE What are the steps to convert a BST to a sorted doubly linked list in-place (no extra space)?
Answer
① In-order traversal (gives sorted order). ② At each node: reuse left pointer as "prev" pointer in DLL, right pointer as "next" pointer. ③ Track previous node; set prev.right = current, current.left = prev. ④ Head = leftmost node. ⑤ After traversal, head.left = tail, tail.right = head for circular DLL (optional). Time: O(n), Space: O(h) recursion only.
