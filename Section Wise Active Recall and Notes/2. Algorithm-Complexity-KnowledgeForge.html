⏱️ Time Complexity — How We Measure Algorithm Speed

💡 You can't use a stopwatch to compare algorithms fairly — different machines give different times. Instead, count the number of primitive operations as a function of input size n.

Two Ways to Analyse an Algorithm

MethodHow It WorksProblem
ExperimentalRun the code, measure actual timeDepends on hardware, OS, compiler — results aren't portable
TheoreticalCount primitive operations as a math function of nSlightly abstract — but hardware-independent and universal
📌 We always use theoretical analysis. It counts primitive operations and expresses the total as f(n), then strips away constants to get Big-O.

What is a Primitive Operation?

A single, indivisible step that takes constant time O(1) regardless of input:

  • Declaration: int x
  • Assignment: x = 5
  • Comparison: i < n
  • Arithmetic: a + b, a * b
  • Array access: A[i]
  • Function call/return

Example 1 — Sum of N Numbers → O(n)

csharp
int Sum(int n)
{
    int total = 0;   // 1 decl + 1 assign
    int i = 1;       // 1 decl + 1 assign

    while (i <= n)   // comparison: runs N+1 times
    {
        total = total + i;  // arithmetic + assign: N times each
        i = i + 1;          // arithmetic + assign: N times each
    }
    return total;
}

Total: 2 + 2 + (N+1) + 2N = 5 + 3N → O(n)

Example 2 — Binary Search → O(log n)

csharp
int BinarySearch(int[] A, int key)
{
    int i = 0;
    int j = A.Length - 1;

    while (i < j)
    {
        int mid = (i + j) / 2;    // each iteration halves the search space
        if (A[mid] == key) return 1;
        else if (key < A[mid]) j = mid - 1;
        else i = mid + 1;
    }
    return -1;
}
💡 Each loop iteration halves the problem: N → N/2 → N/4 → … → 1. The number of halvings is log₂N. That's where O(log n) comes from.

Total: 2 + 2 + (log N + 1) + log N + log N + log N = 5 + 5 log N → O(log n)

Example 3 — Matrix Sum (Nested Loop) → O(n²)

csharp
int MatrixSum(int[,] A, int n, int m)
{
    int total = 0;
    for (int i = 0; i < n; i++)         // outer: runs N times
    {
        for (int j = 0; j < m; j++)     // inner: runs M times per outer
        {
            total = total + A[i, j];
        }
    }
    return total;
}

Total: ≈ 6 + 6N + 4NM. If M = N → 6 + 6N + 4N² → O(n²)

🔬 Asymptotic Analysis — Why Constants Don't Matter at Scale

💡 "Asymptotic" means: what happens as n → ∞? At large inputs, only the fastest-growing term matters. The rest are noise.

The Dominant Term Principle

Consider f(n) = n² + 6n + 5

  • At n = 10: n² = 100, 6n = 60, 5 = 5 — all comparable
  • At n = 1,000: n² = 1,000,000 · 6n = 6,000 · 5 = 5 — n² completely dominates
  • Asymptotically: f(n) ≈ n² → O(n²)

📌 We almost always analyse worst case (Big-O). It's a guarantee — your algorithm will never be slower than this. Best case is rarely relevant in production.

🔴 Big-O Notation — Upper Bound (Worst Case)

Big-O — O(g(n))

f(n) ≤ c · g(n)   for all n ≥ n₀

f(n) grows no faster than g(n). The algorithm takes at most this long.

Formal Definition

  • f(n) = O(g(n)) if there exist constants c > 0 and n₀ ≥ 1 such that f(n) ≤ c · g(n) for all n ≥ n₀
  • f(n) is your actual algorithm cost function
  • g(n) is the upper bound function (e.g. n, n², log n)
  • c is a constant multiplier that makes the inequality work
  • n₀ is the "threshold" — the inequality only needs to hold for large enough n

Proof: f(n) = 5n + 4 is O(n)

Need to show: 5n + 4 ≤ c · n for all n ≥ n₀

  • Choose c = 6, n₀ = 4: check 5(4) + 4 = 24 ≤ 6(4) = 24 ✓
  • For all n ≥ 4: 5n + 4 ≤ 6n (the +4 becomes insignificant) ✓
  • Many valid (c, n₀) pairs exist — always choose the tightest useful one

🟢 Big-Omega — Lower Bound (Best Case)

Big-Ω — Ω(g(n))

f(n) ≥ c · g(n)   for all n ≥ n₀

f(n) grows at least as fast as g(n). The algorithm takes at minimum this long — can't go faster.

Formal Definition

  • f(n) = Ω(g(n)) if there exist constants c > 0 and n₀ ≥ 1 such that f(n) ≥ c · g(n) for all n ≥ n₀
  • The function g(n) is a lower bound — the algorithm cannot be faster than this

Proof: f(n) = 5n + 4 is Ω(n)

  • Need: 5n + 4 ≥ c · n for all n ≥ n₀
  • Choose c = 1, n₀ = 1: 5(1) + 4 = 9 ≥ 1(1) = 1 ✓ (trivially satisfied)
  • Result: f(n) = Ω(n)

🟡 Big-Theta — Tight Bound (Average Case)

Big-Θ — Θ(g(n))

c₁·g(n) ≤ f(n) ≤ c₂·g(n)   for all n ≥ n₀

f(n) is sandwiched between two bounds. The algorithm is exactly this class — both upper and lower bound are the same growth rate.

Formal Definition

  • f(n) = Θ(g(n)) if there exist constants c₁, c₂ > 0 and n₀ ≥ 1 such that c₁·g(n) ≤ f(n) ≤ c₂·g(n) for all n ≥ n₀
  • Θ = Big-O and Big-Ω at the same time
  • Used when best case = worst case (e.g., summing n numbers — you must visit all n)

Proof: f(n) = 5n + 4 is Θ(n)

BoundInequalityConstantsValid from
Upper (Big-O)5n + 4 ≤ 6nc₂ = 6n₀ > 4
Lower (Big-Ω)5n + 4 ≥ 1·nc₁ = 1n₀ ≥ 1

Both hold → f(n) = Θ(n)

📊 Notation Summary — All Three Together

Big-O (Worst Case)

f(n) ≤ c · g(n)

Upper bound. The algorithm is no worse than this. Primary tool for analysis.

Big-Ω (Best Case)

f(n) ≥ c · g(n)

Lower bound. The algorithm is no better than this. Rarely the focus.

Big-Θ (Tight Bound)

c₁·g(n) ≤ f(n) ≤ c₂·g(n)

Both bounds are the same. Used when best = worst case.

Q01DEF What is a primitive operation and why does it matter for time complexity analysis?
Answer
A primitive operation is a single indivisible step that takes constant time O(1) — declaration, assignment, comparison, arithmetic, array access, function call. We count these instead of measuring wall-clock time because the count is hardware-independent. The total count expressed as f(n) is what we convert to Big-O.
Examples: int x = 0 = 1 decl + 1 assign. i < n = 1 comparison.
Q02COMPARE What is the difference between experimental analysis and theoretical analysis of algorithms? Which is preferred and why?
Answer
  • Experimental: run the code and time it. Problem: results depend on hardware, OS, compiler — not portable or comparable.
  • Theoretical: count primitive operations, express as f(n). Hardware-independent, covers all inputs, mathematically comparable.
Theoretical is preferred because it gives a universal result you can compare across algorithms before writing any code.
Q03APPLY Count the primitive operations in this loop and determine its Big-O:
for (int i = 0; i < n; i++) { total += A[i]; }
Answer
  • Declaration int i: 1
  • Assignment i = 0: 1
  • Comparison i < n: N + 1 times
  • Array access A[i]: N times
  • Addition + assignment: N + N = 2N times
  • Increment i++: N times
Total ≈ 2 + (N+1) + 4N = 3 + 5N → O(n)
Q04WHY Why does binary search run in O(log n) and not O(n)?
Answer
Each iteration of binary search halves the remaining search space: N → N/2 → N/4 → … → 1. The number of times you can halve N before reaching 1 is log₂N. So the loop runs log N times, not N times.
Numbers: N = 1024: binary search takes at most 10 steps. Linear search takes up to 1024 steps.
Q05SEQUENCE List the 7 complexity classes in order from fastest to slowest (best performance to worst).
Answer
  1. O(1) — Constant
  2. O(log n) — Logarithmic
  3. O(n) — Linear
  4. O(n log n) — Linearithmic
  5. O(n²) — Quadratic
  6. O(n³) — Cubic
  7. O(2ⁿ) — Exponential (avoid entirely)
Memory trick: "Can Little Nerds Learn Quadratic Cubic Exponents?"
Q06GOTCHA Two separate (non-nested) for-loops each running n times — what is the total Big-O?
Answer
O(n) — not O(n²). Two sequential loops add: O(n) + O(n) = O(2n) = O(n). Only nested loops multiply to give O(n²). The key is whether the loops are inside each other or one after the other.
Rule: Sequential = add. Nested = multiply.
Q07DEF State the formal definition of Big-O notation using mathematical notation.
Answer
f(n) = O(g(n)) if there exist positive constants c > 0 and n₀ ≥ 1 such that:

f(n) ≤ c · g(n)   for all n ≥ n₀

g(n) is the upper bound function. n₀ is the threshold after which the inequality holds. f(n) represents actual algorithm cost.
Q08APPLY Prove that f(n) = 5n + 4 is O(n) by finding valid constants c and n₀.
Answer
Need: 5n + 4 ≤ c · n for all n ≥ n₀

Choose c = 6, n₀ = 4:
Check: 5(4) + 4 = 24 ≤ 6(4) = 24 ✓
For all n ≥ 4: 5n + 4 ≤ 6n (since 4 ≤ n when n ≥ 4) ✓

Result: f(n) = O(n)
Note: Many valid (c, n₀) pairs exist. c=7, n₀=2 also works. Always choose the tightest useful bound.
Q09DEF State the formal definition of Big-Omega (Ω) and what it represents.
Answer
f(n) = Ω(g(n)) if there exist constants c > 0 and n₀ ≥ 1 such that:

f(n) ≥ c · g(n)   for all n ≥ n₀

Big-Ω is the lower bound — the algorithm cannot perform faster than this in the best case. It's the mirror image of Big-O.
Q10APPLY Prove that f(n) = 5n + 4 is Ω(n).
Answer
Need: 5n + 4 ≥ c · n for all n ≥ n₀

Choose c = 1, n₀ = 1:
Check: 5(1) + 4 = 9 ≥ 1(1) = 1 ✓ (trivially satisfied)
5n + 4 ≥ n for all n ≥ 1 since 4n + 4 ≥ 0 always ✓

Result: f(n) = Ω(n)
Q11DEF State the formal definition of Big-Theta (Θ) and what it means about an algorithm.
Answer
f(n) = Θ(g(n)) if there exist constants c₁, c₂ > 0 and n₀ ≥ 1 such that:

c₁ · g(n) ≤ f(n) ≤ c₂ · g(n)   for all n ≥ n₀

f(n) is sandwiched between two scaled versions of g(n). It means Big-O and Big-Ω are the same class — the tight bound. Used when best case = worst case.
Example use: Summing n numbers always takes exactly n iterations — no best/worst variation. So it's Θ(n).
Q12APPLY What is the Big-O of: 5n⁴ + 3n³ + 2n² + 4n + 1? Explain how you got there.
Answer
O(n⁴). Steps:
  1. Identify all terms: n⁴, n³, n², n, 1
  2. As n → ∞, n⁴ grows faster than all others
  3. Drop constants (5) and all lower-order terms
  4. Result: O(n⁴)
Q13COMPARE Compare Big-O, Big-Ω, and Big-Θ: which case does each describe, and which is most useful in practice?
Answer
  • Big-O → worst case (upper bound). Most useful in practice — tells you the guarantee.
  • Big-Ω → best case (lower bound). Rarely the focus — best case is often not representative.
  • Big-Θ → tight bound (average case). Used only when best = worst = same class.
Big-O is the primary tool. When someone says "this algorithm is O(n²)", they mean worst-case.
Q14GOTCHA f(n) = 100n vs g(n) = n². Which is "better" at n=5? Which is better at n=1000? What does this tell us about Big-O?
Answer
  • At n = 5: f(5) = 500, g(5) = 25 → g(n) = n² is faster here
  • At n = 1000: f(1000) = 100,000, g(1000) = 1,000,000 → f(n) = 100n is faster by 10×
This shows that Big-O describes asymptotic behaviour (large n), not small inputs. An O(n) algorithm eventually beats O(n²) — even if O(n²) has a smaller constant at first. Constants don't matter at scale.
Q15WHY Why does asymptotic analysis ignore constant factors?
Answer
As n → ∞, the constant multiplier becomes irrelevant compared to the growth rate. Whether an algorithm does 3n or 100n operations, both are O(n) and both are infinitely better than O(n²) for large enough n. Constants also depend on hardware — ignoring them makes analysis hardware-independent and universally comparable.
Q16SEQUENCE Walk through the steps to convert a raw operation count f(n) = 6 + 6n + 4n² to Big-O notation.
Answer
  1. Write out all terms: 6, 6n, 4n²
  2. Identify the highest-growth term: 4n²
  3. Drop constant multiplier (4): → n²
  4. Drop all lower-order terms (6n and 6)
  5. Result: O(n²)
Q17APPLY A function has the total operation count: 5 + 5 log N. What is its Big-O and what type of algorithm produces this pattern?
Answer
Big-O: O(log n). Dominant term is 5 log N; drop the constant 5 and the additive constant 5. This pattern is produced by algorithms that halve the problem size each iteration — like binary search.
Q18DRAW Sketch the Big-Theta graph for f(n) = 5n + 4. Label the three curves and the n₀ threshold.
Answer
f(n) must lie between c₁·g(n) and c₂·g(n) for all n ≥ n₀. With c₁=1, c₂=6, n₀=4, f(n)=5n+4 satisfies this → Θ(n).
Q19WHY When is Big-Theta appropriate to use? Give a concrete algorithm example.
Answer
Big-Θ is appropriate when best case = worst case = same complexity class. This means no matter what input you give, the algorithm always runs in exactly g(n) time — there's no lucky shortcut possible.
Example: Summing n numbers — you must visit all n values every time. There's no "early exit". Best case = worst case = Θ(n).
Q20APPLY What is the space complexity of an algorithm that declares only a fixed number of integer variables (no arrays)? Explain.
Answer
O(1) — constant space. If memory usage doesn't depend on input size n (just a fixed number of int variables), the total bytes is a constant regardless of how large n gets. The memory doesn't grow with the problem.
Example: int total=0, i=1, n=... → 12 bytes always → O(1).
Q21APPLY An algorithm allocates a 2D array A[n][m] of integers plus 3 int variables. What is its space complexity?
Answer
  • 3 int variables: 3 × 4 = 12 bytes (constant)
  • A[n][m] of int: 4 × n × m bytes
  • Total: 12 + 4nm bytes
Space complexity: O(nm). The constant 12 bytes is dropped; only the input-dependent term remains.
Q22GOTCHA Someone says: "O(log n) is barely better than O(1) — both are small." Is this correct?
Answer
Incorrect. O(1) is truly constant — it takes the same time regardless of n. O(log n) does grow with n, just very slowly. The difference matters: at n = 2³⁰ (~1 billion), O(1) still takes 1 step; O(log n) takes 30 steps. That's fine. But O(n) at the same input takes 1 billion steps. So O(log n) and O(1) are both excellent — but they're not the same.
Q23COMPARE For linear search: state its Ω, Θ, and O — and explain each.
Answer
  • Ω(1): Best case — the target is the first element. Done in 1 comparison.
  • Θ(n): Average case — target is somewhere in the middle; ~n/2 comparisons, which is Θ(n).
  • O(n): Worst case — target is the last element or not present; n comparisons.
Q24WHY Why should exponential O(2ⁿ) algorithms be avoided in practice?
Answer
At n=100, 2¹⁰⁰ ≈ 10³⁰ operations. Even at 10¹⁰ operations/second, that's 10²⁰ seconds — far longer than the age of the universe (~4×10¹⁷ seconds). Exponential algorithms are only feasible for very small inputs (n < 20–25 typically). For any real-world input size, they're completely unusable.
Q25SEQUENCE What are the steps to prove that a function f(n) is Big-Theta Θ(g(n))?
Answer
  1. Choose a candidate g(n) (usually the dominant term of f(n))
  2. Prove upper bound: find c₂ and n₀ such that f(n) ≤ c₂·g(n) for all n ≥ n₀ (Big-O proof)
  3. Prove lower bound: find c₁ and n₀ such that f(n) ≥ c₁·g(n) for all n ≥ n₀ (Big-Ω proof)
  4. Both must hold with valid positive constants
  5. Conclude: f(n) = Θ(g(n))
For f(n) = 5n+4: Upper: c₂=6, n₀=4. Lower: c₁=1, n₀=1. Both hold → Θ(n). ✓
